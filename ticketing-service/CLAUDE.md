# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A unified client-communication / support-ticketing platform, split into two independently-run apps in this one repo:

- `backend/` — **historical**: this directory's code was merged into the monorepo-root `unified-backend/` (as `unified-backend/app/ticketing/`) — see the root `CLAUDE.md`'s "Backend consolidation" section. The layering/domain-model/access-control documentation below still describes the actual code accurately (it moved, the architecture didn't change); only the "Commands" section (run-from-`backend/`, port 8001) is stale — see the corrected version there.
- `frontend/` — React 18 + TypeScript + Vite + Tailwind ("Agent Workspace" UI). Still its own real, independently-run app, unaffected by the backend merge (it now points at `unified-backend`'s port instead of a dedicated one).

This service is one half of a two-service monorepo (`Unified-ticket-management-system/`, sibling `rbac-service/` and `shared_models/` at the repo root) that shares a single Postgres database with independent migration histories: `app/rbac` (see `unified-backend/`) owns `users`/`roles`/permissions and is the sole issuer of auth tokens; `app/ticketing` owns `tickets`, `interactions`, `attachments`, `ticket_audit_logs`, `clients`, `mail_folders`, and `ticket_relations`, and only *reads* the RBAC tables (via the `shared_models` package) to resolve the authenticated caller into a full `User` row (role, category, reporting lines). Never write to `users`/`roles`, and never generate a migration that touches them — both remain true now that both domains are one process, since the two Alembic chains (`alembic_rbac`/`alembic_ticketing`) are still deliberately kept independent (see root CLAUDE.md).

**`unified-backend/alembic_ticketing/env.py`'s `OWNED_TABLES` set must list every table this domain owns**, or `include_object` silently excludes it from `alembic revision --autogenerate -c alembic_ticketing/alembic.ini` — the generated migration then references a table that was never created (e.g. a new FK column pointing at it), which crashes on `alembic upgrade`. This has happened twice already (`mail_folders`, initially missed; `ticket_relations` avoided it only because this was already known) — when adding a new model, add its table name to `OWNED_TABLES` in the same commit as the model, before running autogenerate.

**This frontend has a second, embedded copy.** The entire `frontend/` tree (adapted, not just linked — not a build artifact) also lives at `unified-frontend/src/ticket-workspace/` (formerly `rbac-service/frontend/src/ticket-workspace/` — see the root `CLAUDE.md`'s frontend-consolidation note; imported via a `@tw/*` alias instead of `@/*`), mounted inside the shell's Next.js app as the landing experience for Staff/Team Lead/Account Manager (Super Admin, Site Lead, and Viewer instead see the shell's own admin dashboards — see `unified-frontend/CLAUDE.md`). The two copies are **not** kept in sync automatically — a change to a ticket page/component/API wrapper here (this standalone app) needs to be manually ported to the `unified-frontend/src/ticket-workspace/` copy if it should also appear in the embedded experience, and vice versa. This is the single most common way to introduce a "works in one app, broken in the other" bug in this codebase — always grep the other tree for the same filename before considering a frontend change done.

**A real, currently-standing example of this divergence**: the Mail rich-text composer (`components/mail/RichTextEditor.tsx`, built on `@tiptap/react`) exists **only** in the embedded copy (`unified-frontend/src/ticket-workspace/components/mail/`) — this standalone app has no `components/mail/` directory at all. Don't assume feature parity between the two copies without checking; this one was never ported back (or was embedded-only from the start) and hasn't been reconciled.

## The domain model (client → interaction → ticket)

- **Clients** (`app/models/client.py`) are companies, not individual people — each one is onboarded with a dedicated shared inbox address (`inbox_email`, e.g. `abc@probeps.com`) and an owning **Account Manager** (`account_manager_id`, FK → `users`, `nullable=False`, validated at creation by `ClientService.create` to be an active Account Manager). Inbound email is resolved by matching `to_email` against `clients.inbox_email`, never by who sent it, and routing is always this 1:1 mapping — **never round-robin or least-busy-agent**, in the seed data or at runtime. `ClientResponse.account_manager_active` (computed in `ClientService`, backed by `UserRepository.get_active_account_manager_ids`) flags a "soft-orphaned" client — the FK still points at a real user, but that user's role changed (or they were deactivated) *after* the client was onboarded, and nothing revalidates that automatically. `EmailService.receive_email` logs a `logger.warning` when this happens but still ingests the email normally — never drop/bounce real customer mail over a stale mapping.
- **Every inbound email becomes an `Interaction` first** (`interaction_type="EMAIL"`, `ticket_id=NULL`, `status=PENDING`) — landing in the owning Account Manager's inbox, not auto-assigned to any Staff agent (auto-assignment was removed; there is no "least-loaded agent" logic anywhere in this service anymore). Only when an agent decides it's an actionable issue does it get promoted to a `Ticket`.
- **Threading** is real columns on `interactions`, not payload JSON, because inbox queries filter on them directly: `client_id`, `parent_interaction_id` (self-FK — `NULL` means "this row is a thread root"; set means "this row replies under that root"), `received_at` (mailbox arrival time reported by the transport layer — the SLA clock start, distinct from `created_at` which is just when our backend happened to persist the row; see `app/models/interaction.py`'s column comments for why the distinction matters).
- **Every code path that creates a reply-type interaction must set `parent_interaction_id` to the conversation's thread root**, resolved via `existing_interaction.parent_interaction_id or existing_interaction.interaction_id` (walk up once — roots never chain). Forgetting this is a real bug class here: it happened in `InteractionService.add_reply` (ticket-level replies) and in `EmailService.receive_email`'s "already ticketed" branch, both fixed by threading through the resolved root before creating the row. If you add a new interaction-creating method that represents a reply/follow-up in an email conversation, thread it the same way — `_create_ticket_interaction`'s `parent_interaction_id` param exists for exactly this, and defaults to `None` for the other interaction types (notes, status/priority changes, transfers, claims) that aren't part of the client email thread.
- **Tickets** carry both `client_id` (legacy FK → `users`, nullable, only ever set on tickets created before the client-company model — do not write to it going forward) and `client_company_id` (FK → `clients`, the current one). `ticket_type` is the ticket's **work-specialization category** (Eligibility, AR, Claims, Patient Calling, Payment Posting, PA, Charge Entry, ...) — a plain `String(50)` column, deliberately not a fixed Pydantic or Postgres enum: the category list itself lives in the RBAC-owned `categories` table (`shared_models.models.Category`/`CategoryName`), read here read-only via `GET /categories` to populate the ticket-creation dropdown, so it can grow without a migration in *this* service. Every Staff/Team Lead `User` also has a `category_id` (required at creation time on the RBAC side) — see "Roles" below for how that drives ticket visibility.
- **Outbound replies build a real envelope** (`app/services/email_envelope.py`'s `build_reply_envelope`): From = the client's shared inbox, To = the original sender, `Re:`-prefixed subject (no double-prefixing), a generated `message_id`, References chaining, and the Account Manager auto-CC'd. Stored as `payload.envelope` + `payload.dispatch_status` (`QUEUED`/`NO_RECIPIENT`) on the `REPLY` interaction, then handed to `app/services/outbound_dispatcher.py`'s `OutboundDispatcher` — currently a logging no-op; whichever task owns the actual transport/webhook layer replaces this, not the call sites.
- **Mail metadata (tags/folders/snooze/drafts) lives on the pre-ticket `Interaction`, orthogonal to `status`**: `Interaction.tags` (JSONB `list[str]`, full-replace on write, no join table — same free-form-JSON-over-lookup-table philosophy as `payload`/`Ticket.custom_fields`), `Interaction.folder_id` (nullable FK → `mail_folders`, a global/shared-not-per-user lookup table with just `name`/`created_by`/`created_at` — CRUD is un-audited org config, unlike per-interaction actions), and `Interaction.snoozed_until` (nullable datetime — a snoozed item just fails the `list_inbox` "pending" view's `snoozed_until IS NULL OR snoozed_until <= now()` filter and passes the "snoozed" view's inverse, so it resurfaces automatically on its own with no background job). `Interaction.is_draft` marks a normal REPLY/OUTBOUND row (same shape a sent reply would have) that's saved but never dispatched — see "Drafts" below. All four are deliberately available regardless of ticket/claim state *except* snooze and drafts, which only make sense pre-ticket (see the `_resolve_pending_thread_root`/ticket_id checks in the relevant service methods).
- **Related Tickets is a `Ticket`-level feature, not a Mail one** — a bare pre-ticket Interaction has nothing to relate yet. `ticket_relations` is a self-referential join table (composite PK `(ticket_id, related_ticket_id)`) written **symmetrically**: linking A↔B inserts both `(A, B)` and `(B, A)` in one call (`TicketRelationRepository.create`), so reading "what's related to ticket X" is always a single `WHERE ticket_id = :id` query regardless of which side of the pair X was originally linked from — no `OR`-across-two-columns query needed. `TicketResponse.related_tickets` is populated only by `TicketService.get_by_id` (the detail view), never `list_all`, to avoid an N+1 lookup on every row of the tickets list.

## Roles

RBAC's actual roles are **Super Admin, Site Lead, Account Manager, Team Lead, Staff, Viewer** — there is no role literally named `"Manager"` (an older draft of this system assumed one; if you see `"Manager"` hardcoded anywhere as a role-name string, it's dead/wrong, not a role to design around). `app/services/access_control.py` is the source of truth — re-read it rather than trusting a hardcoded set from memory (including the one below), it has shifted more than once as the org-hierarchy model was refined:
- `AGENT_ROLE_NAMES` — every role except Viewer; who can authenticate as an agent at all (`get_current_agent` 403s anyone else). Site Lead and Account Manager were both missing from this set at points during development, which silently 403'd every ticketing endpoint for real users holding those roles — if a role can't do anything in Ticketing, check this set first.
- `SUPERVISOR_ROLE_NAMES` — Team Lead/Account Manager/Site Lead/Super Admin; who bypasses ownership scoping.
- `ACCOUNT_MANAGER_ROLE_NAME` — the literal role string (`"Account Manager"`) that `clients.account_manager_id` must point at; `ClientService.create` validates against it, and `TicketService._resolve_owned_client_ids` uses it to decide whether the caller's ticket list/detail access is scoped to only their own clients.
- **Category-based ticket routing for Team Lead/Staff is implemented** (`CATEGORY_SCOPED_ROLE_NAMES = {"Team Lead", "Staff"}` in `access_control.py`): each work-specialization category is its own shared pool. `ensure_agent_can_view_ticket` compares `current_user.category.category_name` against `ticket.ticket_type` and 403s a Team Lead/Staff trying to view a ticket outside their own category (a user with no category assigned sees nothing, not everything — the safe failure mode); `TicketService._resolve_category_ticket_types` applies the same scoping to `GET /tickets`' list view. Requires `UserRepository.get_by_id` to eager-load `User.category` (`selectinload`) — a lazy load would crash in this async context. Account Manager/Site Lead/Super Admin are unaffected: Account Manager stays scoped by client ownership (see above), Site Lead/Super Admin remain fully unrestricted.
- **Only Team Lead/Account Manager/Site Lead/Super Admin can reassign a ticket to a different named agent by default** — `ensure_can_reassign_ticket` (`access_control.py`) checks `SUPERVISOR_ROLE_NAMES` first, then falls through to `ensure_has_permission(current_user, "ticket:transfer")` for anyone else, so a Staff member individually granted `ticket:transfer` via a personal permission override (see `unified-frontend/CLAUDE.md`'s "Per-user permission overrides") passes this too — making the seed data's own long-standing "Override-only for Staff" comment actually true, not just aspirational. `claim_ticket` (self-pickup of an *unclaimed* ticket from the shared pool) deliberately has **no** such gate — every agent role, Staff included, can still claim an open ticket for themselves; only *moving a ticket to a specific other agent* is restricted. Don't conflate the two when adding a similar check elsewhere.
- **`ensure_agent_can_act_on_ticket`** (`access_control.py`) restricts most ticket-mutating actions (internal notes, replies, status change) to the ticket's actually-assigned agent, on top of `ensure_agent_can_view_ticket`'s category gate — supervisors bypass this by role, same as everywhere else. For your own assigned ticket, it checks `ticket:editown_ticket`; for anyone else's, `ticket:editother_ticket` (globally, or scoped to that one ticket — see below), or having an approved, not-yet-expired **edit-access grant** for that specific ticket (see "Edit access requests" below) — a third, independent way to act on a ticket you're not assigned to. `change_priority` is a deliberate exception to this whole gate — it's checked by `ensure_has_permission(current_user, "ticket:change_priority")` alone, so any permission holder (by role or override) can change priority on any ticket in their visibility scope, not just their own.
- **`ticket:editown_ticket`** (default for every role, including Staff) and **`ticket:editother_ticket`** (default for Super Admin/Site Lead/Account Manager/Team Lead, not Staff — see `unified-backend/scripts/rbac_seed/seed.py`'s `DEFAULT_ROLES`) replaced a single, older `ticket:edit_ticket` permission that conflated "can work my own ticket" with "can work anyone's" — splitting them made it possible to grant the second capability narrowly. `editother_ticket` is the same permission `ensure_can_review_edit_access` requires to approve/reject someone else's edit-access request (see "Edit access requests" below). Checked via the JWT's `permissions` claim (`has_permission`/`ensure_has_permission`) for a blanket grant, never a role-name allowlist, so granting or revoking it (by role default or a personal override) takes effect the next time the affected user's token refreshes, not immediately. **`editother_ticket` can also be scoped to one specific ticket** — see `has_permission_for_ticket` and the JWT's separate `scoped_permissions` claim (permission name → ticket ids), consumed via the shell app's Permission Request workflow when a Staff member needs temporary access to one teammate's ticket rather than every ticket in scope (see `unified-frontend/CLAUDE.md`'s "Permission requests" section for how the request/approve flow and the two ticket-picker endpoints work). A scoped grant never appears in the flat `permissions` list — only in `scoped_permissions` — so it can't be mistaken for a blanket grant by a plain membership check.
- **`GET /agents?category=<name>`** (`app/api/agent.py`, `UserRepository.list_active_staff_by_category`) scopes the Staff picker to one work-specialization category — used by the ticket "Assign to Staff"/"Transfer Agent" modal so a Team Lead only sees their own team's Staff, not every active Staff member company-wide. Omitting `category` returns everyone (unfiltered), which is still what a few other call sites want.

## Commands

```bash
python -m venv .venv && .venv\Scripts\activate                            # Windows, at the monorepo root
pip install -r requirements.txt

Before the backend merge this needed to be byte-identical to a *separate* RBAC `.env`'s value — now there's only one `.env` to get right.
**Backend — this repo's `backend/` directory no longer runs on its own.** Its code was merged into the monorepo-root `unified-backend/` (as `app/ticketing/`, alongside RBAC's own `app/rbac/`) — see the root `CLAUDE.md`'s "Backend consolidation" section. Run it from there instead:

.venv\Scripts\activate                                           # Windows

alembic -c alembic_ticketing/alembic.ini upgrade head             # this service's own migration history/versions dir

```
Every route path is byte-identical to the old standalone `:8001` service (unprefixed, exactly as before) — only the port and process changed. Requires a single `.env` in `unified-backend/`: `DATABASE_URL` (async) + `ALEMBIC_DATABASE_URL` (sync, for Alembic), plus `JWT_SECRET_KEY`/`JWT_ALGORITHM` — no longer needs to match a *separate* RBAC `.env`, since there's only one file now. `JWT_SECRET_KEY` has no default; the app fails fast at boot if it's unset.

**Settings are cached for the process lifetime** (`app/core/config.py`, `get_settings()` is `@lru_cache`d) and `uvicorn --reload` only restarts on Python file changes — editing `.env` while the server is already running does nothing until you kill and restart the process. If a config-dependent error (missing storage credentials, wrong JWT secret, etc.) persists after editing `.env`, restart the server before debugging further; don't assume the code is wrong.

**Windows gotcha — `--reload` is unreliable in this dev environment**: uvicorn's WatchFiles-based reloader has silently failed to pick up code changes more than once during development here — it logs `Reloading...` and keeps serving stale code, which reads exactly like a bug in the changed code itself. If a fix "isn't taking effect" after a save, don't trust `--reload` — kill the whole process tree (`taskkill //F //T //PID <pid>`, not just the reported uvicorn PID, since `--reload` forks a child) and restart without `--reload`.

## Backend architecture

**Path note**: every `app/...` path below is relative to `unified-backend/app/ticketing/`, not this repo's own (now-empty) `backend/`. `app/main.py` (the one registering every router below) lives at `unified-backend/app/main.py` and mounts these routers unprefixed (`GET /tickets`, not `GET /api/v1/tickets` — the `/api/v1` prefix is `app/rbac`'s alone).

Strict layering, always in this direction — routers never touch the DB or construct responses directly:

`app/api/*` (routers, one file per resource: `email`, `agent`, `client`, `inbox`, `ticket`, `interaction`, `attachment`, `category` — all registered in `unified-backend/app/main.py`) → `app/services/*` (business logic, orchestrates repositories) → `app/repositories/*` (the only layer that runs SQLAlchemy queries) → `app/models/*` (ORM) / `app/schemas/*` (Pydantic request/response).

**Auth is real JWT, not a query param.** `app/dependencies/auth.py`'s `get_current_user` decodes an RBAC-issued access token (`app/auth/jwt.py`, decode-only — this service never issues tokens, don't add `create_access_token` here) and re-resolves the user from the shared `users` table on every request (so a deactivated account is caught immediately, not just at next login); `get_current_agent` additionally requires `role.name in AGENT_ROLE_NAMES`. There is no more `agent_name` query-param "acting as" model — if you see that pattern referenced anywhere (old comments, stale skill docs), it predates the real-auth migration and no longer applies.

**Adding a new ticket-mutating action** (status/priority change, transfer, claim, resolve, etc.) means touching all of these in the same shape — `InteractionService` in `app/services/interaction_service.py` has one method per action (`add_reply`, `change_status`, `change_priority`, `transfer_agent`, `claim_ticket`) that all follow the identical recipe: fetch-or-404 → `ensure_agent_can_view_ticket` (category gate) and/or `ensure_agent_can_act_on_ticket` (assigned-agent-or-supervisor-or-permission-or-edit-access-grant gate — see "Roles" above) → optionally a role/permission-restriction gate (`ensure_can_reassign_ticket`, or a direct `ensure_has_permission(current_user, "ticket:...")` call like `change_priority`'s, for actions that are a real authorization rule rather than just a visibility scope) → resolve the acting agent via `AuditLogService.resolve_agent_actor(current_user)` (sync, takes the already-verified `User` from the route's `Depends(get_current_agent)` — no lookup, no `agent_name` string) → mutate the `Ticket` via `TicketRepository.update` (or a dedicated conditional-update method, e.g. `claim`'s race guard) → write an `Interaction` row via the shared `_create_ticket_interaction` helper → write an `AuditLog` row via `AuditLogService.log_event`. Copy the closest existing method rather than inventing a new shape. See the **add-ticket-action** skill.

**Caveat, since this bit the SLA feature below once**: `change_status` and `change_priority` specifically no longer write an `Interaction` row at all — `STATUS_CHANGE`/`PRIORITY_CHANGE` are two of the event types `app/services/audit_to_interaction.py`'s `SYNTHESIZABLE_EVENT_TYPES` now synthesizes a Timeline display row for directly from the `ticket_audit_logs` row instead (same treatment for `AGENT_TRANSFERRED`, `TICKET_CLAIMED`, and the three `EDIT_ACCESS_*` events — see that module's own docstring). Any new hook into `change_status`/`change_priority` has no real `Interaction` row to reference anymore for that specific call — don't assume one exists (pass `None` for anything expecting an interaction id from this method, rather than reading a variable that's no longer created).

**A second, related pattern for actions on a pending pre-ticket `Interaction`** (no `Ticket` row exists yet — the shared inbox pool): `claim_interaction` ("Assign to me"), `archive_interaction` ("Informational/Archive"), `snooze_interaction`/`unsnooze_interaction`, `set_interaction_tags`, and `set_interaction_folder` in the same `InteractionService` mutate the `Interaction` row directly via a matching repository method, gated by `_ensure_can_act_on_pending_interaction` (own-client-scope-or-supervisor, not `ensure_agent_can_view_ticket`) rather than going through `_create_ticket_interaction`/`TicketRepository` at all. `claim`/`archive`/`snooze`/`unsnooze` are atomic conditional-UPDATE guards (mirroring `TicketRepository.claim`'s race-guard shape but keyed on `Interaction.claimed_by`/`.status`/`.snoozed_until` instead of `Ticket.agent_id`) since they're contested "only one winner" actions; `set_tags`/`set_folder` are plain updates since tagging/filing isn't contested and stays available regardless of ticket/claim state (unlike claim/archive/snooze, which stop being valid once the item leaves the pending pool — snooze and claim both 400 if `ticket_id` is already set).

**Drafts extend the pending-Interaction pattern with an upsert-per-thread twist**: a draft is a normal REPLY/OUTBOUND `Interaction` (`parent_interaction_id` set to the thread root) with `is_draft=True`, never dispatched until explicitly sent — "one active draft per thread per agent" means `InteractionRepository.get_draft(root_id, performed_by)` is always at most one row, and `save_draft` upserts it (update in place if found, `create` via `InteractionCreate(..., is_draft=True)` otherwise) rather than ever creating a second draft on the same thread. All three draft actions (`save_draft`/`send_draft`/`discard_draft` in `InteractionService`) start by calling `_resolve_pending_thread_root(interaction_id)` — a shared helper that walks any id within the thread (root, a reply, or the draft itself) up to the root and 400s if that thread has already become a ticket — since a client might hand in the draft's own id, not the root's. `send_draft` deliberately does **not** duplicate the envelope/dispatch/audit logic: it hands the saved text (now also Cc/Bcc, see below) straight to `add_interaction_reply`, so there is exactly one code path that turns text into a real sent reply. `OpenEmailResponse.draft_message` (populated by `OpenEmailService.get_email_details`, which now optionally takes `current_user`) lets the reply composer prefill/resume an existing draft when a thread is opened.

**Drafts now carry Cc/Bcc and attachments, not just message text** (added for the embedded frontend's auto-saving Draft composer — see `unified-frontend/CLAUDE.md`'s "Mail v2" section for the UI side). Nothing here needed a migration or a new column — a draft's `payload` is still the same flexible JSONB dict, just with two new keys: `{"message": ..., "cc": [...], "bcc": [...], "dispatch_status": "DRAFT"}`. `DraftSaveRequest`/`DraftResponse`/`OpenEmailResponse` (`draft_cc`/`draft_bcc`/`draft_attachments` fields) were extended to match; `InteractionRepository.update_draft_message` now merges `cc`/`bcc` alongside `message`. `send_draft` reads `cc`/`bcc` back out of `draft.payload` and passes them into `InteractionReplyRequest` (which already supported `cc`/`bcc` for a normal reply — drafts just weren't threading them through before).

**Draft attachments reuse the existing interaction-scoped upload machinery — no new storage capability, just a new seam exposing it.** `AttachmentService.validate_and_store_files(files, interaction_id)` was already generic (keyed on `interaction_id`, never `ticket_id` — `Attachment` has no `ticket_id` column at all; ticket association is always indirect via `Attachment.interaction_id -> Interaction.ticket_id`), proven by two existing pre-ticket callers: inbound email intake (`EmailService.receive_email`) and Compose (`compose_email`). `POST /inbox/{interaction_id}/draft/attachments` (new route in `api/inbox.py`) is the third: it resolves/creates the current user's draft via a new `InteractionService._get_or_create_draft` helper (shared with `save_draft`'s upsert logic — creates an empty-message draft row if the user attaches a file before typing/saving any text) and calls the same `validate_and_store_files`. This is also why "Attach Files" now works on a reply to a thread that isn't a ticket yet — previously the only exposed upload route was the ticket-scoped `POST /tickets/{id}/attachments`, so pre-ticket attachments had no route at all, not just a disabled button.

**`send_draft` reassigns the draft's attachments onto the newly-created reply before deleting the draft row** (`AttachmentRepository.reassign_interaction(old_interaction_id, new_interaction_id)`, a plain bulk `UPDATE`) — otherwise files uploaded against the draft's `interaction_id` would be orphaned the moment `delete_draft`'s hard delete removes that row. **`discard_draft` now also deletes the draft's attachments** (both the storage objects via `StorageService.delete` and the `Attachment` rows) before discarding the draft itself, since a discarded draft has no surviving row for anything to reference them through afterward. Both `send_draft`/`discard_draft`/`save_draft`'s routes now construct their `InteractionService` with `attachment_repository`/`storage_service` wired in (previously only `open_email`/`compose_email` needed them).

**Sent and Drafts are separate `GET /inbox/sent` / `GET /inbox/drafts` endpoints, not `view=` values on `GET /inbox`** — a sent reply or a draft is itself a thread *child* (real column `parent_interaction_id` set), while every `view` on `GET /inbox` filters `parent_interaction_id IS NULL` (roots only); overloading the `view` enum to also mean "thread children, scoped to `performed_by` instead of `account_manager_id`" would strain that one query past its actual shape. Both endpoints resolve their listed rows' missing `client_name`/`subject` (a REPLY interaction's own payload is just `{"message": ..., "envelope": {...}}`, not an `EmailPayload`) via a batched `InteractionRepository.list_by_ids` lookup of each row's thread root — see `InboxService.get_sent`/`get_drafts` for the identical resolution pattern in both.

**`OpenEmailService.get_email_details` (`GET /inbox/{id}`) is written to never crash on a non-root id**: it resolves `parent_interaction_id` up to the true root before doing anything else (a reply/draft/sent-item's own id is a valid, expected input — the Sent/Drafts frontend adapters deliberately pass `root_interaction_id ?? interaction_id` so this is the normal path, not an edge case), and if the resolved row's `payload` still doesn't validate as `EmailPayload` (a genuinely rootless reply — legacy data from before the threading rule was enforced everywhere), it degrades to a synthesized `EmailPayload(subject="(no subject)", body=<the reply's own "message" field>)` instead of letting the raw `pydantic.ValidationError` surface as a 500.

**Interaction vs. AuditLog — two deliberately separate logs, not one:**
- `interactions` table (`app/models/interaction.py`) is the business-facing ticket **timeline** agents read day to day (emails, replies, notes, status/priority changes, transfers, claims, attachment uploads). `interaction_type` is a free-form string column, not a DB enum — adding a new interaction type needs no migration. Rows can be soft-deleted (`is_visible=False`) via the hide endpoints, which the audit trail deliberately never reflects.
- `ticket_audit_logs` table (`app/models/audit_log.py`, named to avoid colliding with an unrelated `audit_logs` table already in the shared DB) is an immutable, compliance-grade record with no `update()`/`delete()` in its repository at all. `entity_type`/`event_type`/`actor_role` **are** native Postgres enums (`SQLEnum` in the model).
- The Account Manager inbox (`GET /inbox`) is itself just a filtered query over `interactions` (`app/repositories/interaction_repository.py`'s `list_inbox`: thread roots only — `parent_interaction_id IS NULL`, `interaction_type == "EMAIL"` — filtered by `view` (`pending`/`replied`/`ticketed`/`archived`/`snoozed`/`all`), optionally to one custom folder (`folder_id`, orthogonal to `view` — composes as `view=all&folder_id=X` to browse a folder regardless of status), and optionally scoped to one Account Manager via a join against `clients.account_manager_id`) — there's no separate inbox table. `archived` (`status == IGNORED`) is the "Informational/Archive" reviewer decision's view — `InteractionStatus.IGNORED` sat unused for a while before this, so adding it needed no enum/migration work, only the new `view` branch and the FastAPI route's `Query(pattern=...)` regex update. `snoozed` is the inverse of `pending`'s own `snoozed_until` filter (see above). `GET /inbox/sent` and `GET /inbox/drafts` are intentionally separate endpoints, not additional `view` values — see the Sent/Drafts paragraphs above.

**Postgres-enum migration gotcha**: `AuditEventType` (and `AuditEntityType`, `ActorRole`, `TicketStatus`, `TicketPriority`, `InteractionStatus`, `InteractionDirection`) are Python `str, Enum` classes mapped via `SQLEnum(..., name="...")`, which SQLAlchemy renders as a native Postgres `ENUM` type. Adding a new member to the Python enum is **not enough** — the deployed Postgres type must also be widened, or writing that value crashes with `InvalidTextRepresentationError`. Always follow up with an Alembic migration doing `op.execute("ALTER TYPE <enum_name> ADD VALUE IF NOT EXISTS '<VALUE>'")` (downgrade is a no-op — Postgres can't drop enum labels). See the **add-postgres-enum-value** skill, and `backend/alembic/versions/7a2d4e9f1c3b_add_email_received_audit_event_type.py` / `4e6b8a1d3c5f_add_ticket_resolved_audit_event_type.py` for the pattern. This migration must run standalone (no other DDL in the same transaction) and, if enqueued in the same PR, generally needs to be applied before code that writes the new value goes live. `ticket_type` (see "The domain model" above) is the deliberate counter-example — its value list lives in a lookup table (`categories`), not a Postgres enum, precisely so it doesn't need this treatment.

**Access control, JWT verification, and the audit actor model**: this service is a token *verifier only* — it never issues tokens. RBAC is the sole issuer (HS256, shared `JWT_SECRET_KEY`). `app/auth/jwt.py` is decode-only (deliberately has no `create_*` function). `app/dependencies/auth.py` has the two route dependencies used everywhere: `get_current_user` (decodes the bearer token, re-loads the `User` from the shared `users` table via `UserRepository.get_by_id` on every request — not trusting the token's embedded claims blindly, since they can be stale up to the token's TTL — 401s on invalid/expired/missing, 403s if `is_active` is `False`) and `get_current_agent` (wraps it, additionally 403s unless the role is in `AGENT_ROLE_NAMES`). There is no local login/signup/refresh endpoint in this service at all — those only exist on RBAC. `ensure_agent_can_view_ticket` (`access_control.py`) is the single view-gating chokepoint, called from `ticket_service.py`, `interaction_service.py`, and `attachment_service.py` alike — see "Roles" above for its category-scoping logic.

Separately, `AuditLogService.resolve_agent_actor(current_user: User | None)` (`app/services/audit_log_service.py`, synchronous — no DB lookup, since the caller already resolved and validated the user via the JWT dependency) turns that `User` into the `(actor_id, actor_name, actor_role)` triple written onto every audit row — `actor_name`/`actor_role` are snapshotted at write time (not joined at read time) so the trail doesn't change retroactively if a user is renamed or removed. `current_user=None` becomes `actor_role=SYSTEM`, reserved strictly for genuinely-automatic writes with no HTTP caller at all (e.g. auto-assignment inside `EmailService.receive_email`) — an authenticated request can no longer produce an unresolvable identity, since a request without a valid token never reaches the service method at all (it 401s at the dependency).

Mutating routes use `current_user = Depends(get_current_agent)`; read routes use `Depends(get_current_user)` (any authenticated user, including Viewer) and let `ensure_agent_can_view_ticket`'s role check do the actual gating. There is no `agent_name` query/form/path param anywhere in the API anymore — `GET /agents/{agent_name}/inbox` was renamed to `GET /agents/me/inbox`, resolving identity from the token server-side. `POST /emails/incoming` (the inbound-email webhook) is the one deliberately unauthenticated route, since it's service-to-service (N8N), not user-facing.

**Permission-based enforcement layered on top of role-name checks**: `access_control.has_permission`/`ensure_has_permission`/`has_permission_for_ticket` read the `permissions`/`scoped_permissions` claims RBAC embeds in the JWT (threaded onto the resolved `User` as transient, non-persisted attributes in `dependencies/auth.py` — same pattern as `TicketService._attach_names` — both defaulting to empty if the claim is absent/stale rather than crashing) — a decode-only check, never a fresh call back to `app.rbac`, matching this domain's verify-only design (now a module-boundary convention, not a network one — see root CLAUDE.md). This is a *second*, finer-grained authorization dimension layered on top of the existing role-name sets (`AGENT_ROLE_NAMES`/`SUPERVISOR_ROLE_NAMES`/etc.), not a replacement for them. Current real call sites: `change_priority` (requires `ticket:change_priority`), `ensure_can_reassign_ticket`'s Staff fallback (requires `ticket:transfer`), and `ensure_agent_can_act_on_ticket`'s split (requires `ticket:editown_ticket` for your own ticket, `ticket:editother_ticket` — globally or scoped to that one ticket, via `has_permission_for_ticket` — for anyone else's, or an approved edit-access grant as a third option; see "Roles" above and "Edit access requests" below). Revoking a permission (role-wide, or via an rbac-service personal override) doesn't invalidate a token already issued — the holder keeps it until their token naturally expires or refreshes, the same pre-existing trade-off role changes already have.

**Edit access requests** — letting more than one person work a ticket at once: `ticket_edit_access_requests` (`app/models/ticket_edit_access_request.py`) is a per-ticket, per-requester request/approve/reject workflow for anyone who isn't the assigned agent and doesn't hold `ticket:editother_ticket` outright. `EditAccessService` (`app/services/edit_access_service.py`) — `request_access`/`approve`/`reject`/`list_for_ticket` — gates review actions with `ensure_can_review_edit_access` (view access + `ticket:editother_ticket`); a partial unique index (`WHERE status = 'PENDING'`) allows only one open request per ticket+requester at a time, and `has_active_grant` (status `APPROVED`, `expires_at` `NULL` or in the future) is what `ensure_agent_can_act_on_ticket` consults to let an approved requester through. Every transition writes **both** an `Interaction` (`EDIT_ACCESS_REQUESTED`/`_APPROVED`/`_REJECTED`, visible on the ticket's own timeline) and a central `AuditLog` row (same three new `AuditEventType` values, added via the Postgres-enum-widening migration pattern above) — see "Interaction vs. AuditLog" for why both exist. Routes live under `POST/GET /tickets/{id}/edit-access*` in `api/ticket.py`. Frontend: `components/ticket/EditAccessPanel.tsx` (mirrored into the RBAC-embedded copy per the dual-copy rule) renders on the ticket detail page — a "Request Edit Access" button for anyone without access, a pending-approval state for the requester, and an approve/reject queue for anyone holding `ticket:editother_ticket`.

This workflow is deliberately separate from — and was left completely untouched by — the shell app's newer ticket-*scoped* Permission Request flow (see `unified-frontend/CLAUDE.md`'s "Permission requests" section), which grants `ticket:editother_ticket` itself for one specific ticket via an `rbac`-owned `UserPermissionOverride.scope_ticket_id`, reviewed by whoever holds `permission:override_grant` + `editother_ticket` (typically Account Manager/Site Lead/Super Admin), rather than this table's own request/approve/reject cycle reviewed by anyone holding `editother_ticket`. Both mechanisms end up satisfying the same `ensure_agent_can_act_on_ticket` check from different data sources (`has_active_grant` here vs. `has_permission_for_ticket`'s `scoped_permissions` claim there) — a ticket can have an approved grant from either or both without conflict.

**SLA workflow (First Response + Resolution)** — two independent clocks (`app/models/{first_response_sla,resolution_sla,resolution_sla_pause_interval,sla_policy,sla_breach_notification}.py`), orchestrated through `SLAService` (`app/services/sla_service.py`, built via the `build_sla_service(db)` factory every hook/route uses) rather than scattered logic at each call site. **First Response** starts the instant a genuinely new email thread root arrives (`EmailService.receive_email`, only when `parent_interaction_id` stays `None` — a reply threading onto an existing pending item or ticket never gets a second clock) and completes via whichever of the four triage actions handles it (`archive_interaction`→`ARCHIVED`, `add_interaction_reply`→`REPLIED`, `attach_to_existing_ticket`→`ATTACHED_TO_TICKET`, `create_ticket_from_interaction`→`TICKET_CREATED`), each calling `SLAService.complete_first_response_clock` — a silent no-op if no clock exists, since SLA bookkeeping must never block the underlying business action. **Resolution** starts when a ticket is created and pauses/resumes off `change_status`'s `WAITING_FOR_CLIENT` transitions (the sole status chokepoint — see the caveat above) plus a second, independent customer-driven resume path inside `EmailService.receive_email`'s already-ticketed branch: a ticket can sit at `status=WAITING_FOR_CLIENT` while its Resolution clock is already `RUNNING` again if the customer replied before an agent got around to changing the status back — always read the clock's own status for SLA purposes, never infer it from `Ticket.current_status`. Entering `CLOSED` (never `RESOLVED`) is the only transition that completes the Resolution clock, and is now gated to `SUPERVISOR_ROLE_NAMES` via the new `access_control.ensure_can_close_ticket` — before this feature, any agent including Staff could close any ticket through `change_status`; this is the precedent if you ever need to add a similar role gate to another status/priority action.

`due_at` is a single mutable timestamp, shifted forward by the exact pause duration on every resume (`ResolutionSLARepository.compute_resumed_due_at`/`compute_reshifted_due_at`, both pure functions factored out specifically for unit testing without a DB — see `tests/test_sla_clock_math.py`) rather than an accumulated-elapsed-time model, so breach detection stays a trivial indexed `WHERE status = 'RUNNING' AND due_at < now()` query. Targets (minutes per `TicketPriority`, both clocks) live in `sla_policies` — 3 seeded rows (HIGH/MEDIUM/LOW), admin-editable via `PATCH /sla/policies/{id}`, gated to a new `sla:manage_policies` permission (Site Lead/Super Admin only by default in `scripts/rbac_seed/seed.py` — deliberately narrower than `SUPERVISOR_ROLE_NAMES`, since these are contractual/company-wide settings, not per-team config).

Breach detection is a **periodic sweep, not an in-process scheduler** — this codebase never had a background-job mechanism before this feature, and still doesn't run one inside the web process (would double-fire if `unified-backend` is ever scaled to more than one instance). `POST /internal/sla/sweep` (`app/api/sla_internal.py`) is hit by a separate Render Cron Job (`render.yaml`'s `sla-sweep` service, `scripts/sla_sweep_ping.py`, every ~2 minutes), protected by a shared-secret header (`X-SLA-Sweep-Secret`, `secrets.compare_digest`, checked against `Settings.sla_sweep_shared_secret` — no default, fails fast like `jwt_secret_key`) rather than JWT, since there's no user behind a cron tick. `SLASweepService` classifies every active clock's `elapsed_fraction` (derived purely from `due_at` and the policy target — never touches `started_at`/pause history at sweep time, the whole payoff of the shifting-`due_at` design) against three thresholds (AT_RISK 80%, BREACHED 100%, ESCALATED 150%) and records each via `SLABreachNotificationRepository.try_record`'s `INSERT ... ON CONFLICT (clock_type, clock_id, threshold) DO NOTHING` — only an actual insert fires the real `NotificationService.notify(...)` call, so the sweep is safe to run repeatedly or overlapping with no application-level lock. In-app notifications only for v1 — `OutboundDispatcher` (real email transport) is still a logging no-op, so there's no email escalation channel to hook into yet.

A manual override exists for business exceptions outside the customer-wait model: `POST /tickets/{id}/sla/pause` and `/resume`, following the usual add-ticket-action recipe (writes a real `SLA_PAUSED`/`SLA_RESUMED` `Interaction` row + an `AuditLog` row via new `SLA_MANUALLY_PAUSED`/`SLA_MANUALLY_RESUMED` `AuditEventType` values, added the standard Postgres-enum-widening way), gated the same as closing a ticket (`ensure_can_override_sla`, supervisor-only). **No frontend work exists for this feature at all yet** in either frontend copy: `SLA_PAUSED`/`SLA_RESUMED` interaction types and the sweep's own `SLA_BREACH_DETECTED`/`SLA_ESCALATED` audit events have no `lib/interactionMeta.ts`/`lib/auditLogMeta.ts` entry, so they render with the generic bullet/JSON-dump fallback described above rather than a proper icon/label; there's no SLA countdown/badge/pause-button anywhere on the ticket detail page, and no policy admin screen. The Dashboard's existing "SLA Risk" stat card predates this feature entirely and is an unrelated client-side heuristic (`hoursSince(ticket.updated_at) >= 24`) — it will not reflect these real clocks; don't use it to demo or debug this feature.

**Storage abstraction**: `app/storage/base.py` defines `StorageService`; `supabase_storage.py` (talks to Supabase Storage's REST API directly via `httpx`, using the project URL + service_role key — not the `supabase` Python package, and not S3-compatible) and `s3_storage.py` are the two implementations, selected at runtime by `STORAGE_BACKEND` (`app/storage/__init__.py`, default `"supabase"`). A misconfigured backend raises `StorageConfigurationError` ("STORAGE_BACKEND=supabase requires SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY to be set"), caught by a global handler in `app/main.py` that returns a clean 503 instead of a raw 500 — don't add another try/except around storage calls, extend that handler if new failure modes need surfacing. Remember the `@lru_cache` gotcha above: adding these vars to `.env` while the backend is already running has no effect until it's restarted.

## SLA & Escalation

Two independent per-ticket clocks, both plain rows keyed 1:1 to a `Ticket` (`resolution_slas`/`first_response_slas`), not columns on `Ticket` itself:

- **First Response SLA** (`app/models/first_response_sla.py`, `FirstResponseSLARepository`) starts when a ticket's founding interaction lands and completes on the first agent reply (`SLAService.complete_first_response_clock`, called from `InteractionService.add_reply`/`add_interaction_reply`/`InboxTicketService.create_ticket_from_interaction`/`attach_to_existing_ticket` — anywhere the first outbound reply or ticket-creation-that-implies-a-response can happen).
- **Resolution SLA** (`app/models/resolution_sla.py`) starts at ticket creation, **pauses** while `current_status == WAITING_FOR_CLIENT` and **resumes** when it leaves that status (`InteractionService.change_status`, via `SLAService.pause_resolution_clock`/`resume_resolution_clock`), **reshifts** its `due_at` forward by the exact pause duration on resume (not an accumulated-elapsed-time recompute — this is what keeps the sweep's own query a cheap, indexed `WHERE status = 'RUNNING' AND due_at < now()`), and **completes** only on a transition into `CLOSED` (`SLAService.complete_resolution_clock`) — entering `RESOLVED` deliberately does not complete it. A priority change reshifts `due_at` via `reshift_resolution_clock_for_priority_change` (`change_priority`). Attaching a pending email to an existing ticket (`InboxTicketService.attach_to_existing_ticket`) creates-or-resumes the clock via `create_or_resume_resolution_clock` (covers a ticket that somehow never had one, or one that was paused). **`SLAClockStatus`** (`PENDING`/`RUNNING`/`PAUSED`/`COMPLETED`, shared by both clock tables rather than split into two enums) is a native Postgres enum (`sla_clock_status_enum`) — see the Postgres-enum-migration gotcha above before adding a value.
- **Target minutes per priority live in `SLAPolicy`** (`sla_policies` table, one row per `TicketPriority`), not hardcoded — a policy change takes effect for every future evaluation with no code change, though it does not retroactively reshift already-running clocks' `due_at`.
- **`SLASweepService.run_sweep()`** (`app/services/sla_sweep_service.py`, invoked via `POST /internal/sla/sweep`, shared-secret-protected — see `app/api/sla_internal.py`'s `verify_sla_sweep_secret`, constant-time `secrets.compare_digest` against `settings.sla_sweep_shared_secret`, no default, fails fast at boot if unset) is the one periodic background pass: batch-fetches every active clock of both types, computes each one's elapsed fraction against four thresholds (`HALF_ELAPSED`/`AT_RISK`/`BREACHED`/`ESCALATED` — `compute_elapsed_fraction`/`thresholds_reached` in `sla_service.py`/`sla_escalation_rules.py`, both pure functions, unit-tested in `tests/test_sla_clock_math.py`), records every newly-crossed `(clock_type, clock_id, threshold)` triple in one batched idempotency-ledger insert (`SLABreachNotificationRepository.try_record_many`, `INSERT ... ON CONFLICT DO NOTHING ... RETURNING` — a threshold notifies at most once, ever), then fires both an in-app notification (always) and a real outbound email (falls back to logging-only until `smtp_host` is configured) only for the genuinely-new crossings, each isolated in its own `db.begin_nested()` savepoint so one entry's failure can't affect another's in the same run. **Recipient resolution is table-driven** (`sla_escalation_rules.py`'s `RESOLUTION_RULES_CLAIMED`/`RESOLUTION_RULES_UNCLAIMED`, `resolve_recipients`) — who gets notified at each threshold for a claimed vs. unclaimed ticket is declared once, there, not hardcoded if/elif in the sweep itself; `tests/test_sla_escalation_rules.py` guards that every threshold has a rule entry and every referenced role has a resolver, and that the owning Account Manager is never dropped from any tier. Triggered externally by a scheduled GitHub Actions workflow, not an in-process scheduler or a Render Cron Job — see root `CLAUDE.md`'s Deployment section for why and the required repo secrets.

**A separate, deliberately-independent internal escalation workflow** (`TicketEscalation` model, `app/services/escalation_service.py`) rides on top of the Resolution SLA's threshold crossings but is architecturally forbidden from ever touching that clock's own `started_at`/`due_at`/`status` — this is the one invariant `tests/test_escalation_service.py` exists to guard, reloading the Resolution SLA row after every escalation mutation in every test and asserting it's byte-identical. `EscalationLevel` (`TEAM_LEAD → MANAGER → SITE_LEAD`, native enum `ticket_escalation_level_enum`) is the ownership chain; `EscalationStatus` (`ACTIVE`/`ACKNOWLEDGED`/`CLOSED`, native enum `ticket_escalation_status_enum`) is the row's own lifecycle. `auto_escalate_if_needed` creates one at `TEAM_LEAD` on a ticket's first `BREACHED`/`ESCALATED` crossing (no-op if one's already active, manual or automatic); `manual_escalate` does the same on demand (requires `ticket:escalate`, 400s if already actively escalated — never a second parallel chain); `acknowledge` (403s for anyone but the resolved owner) stops auto-advance at the current level; `evaluate_overdue` (run at the end of the same sweep tick, not a second scheduler) auto-advances an un-acknowledged escalation past its `ack_due_at`; `close_for_ticket_resolution` closes the escalation (not the SLA clock — that's `SLAService`'s own job, called separately) when a ticket resolves. `SLA_ESCALATED` (the Resolution SLA's own notification-ladder tier, written by the sweep) and `ESCALATION_CREATED`/`_ACKNOWLEDGED`/`_ADVANCED`/`_CLOSED` (this workflow's own events) are deliberately distinct `AuditEventType` values — don't conflate them when adding a new call site.

**Frontend**: `SlaCard.tsx`/`SlaTimeline.tsx`/`SlaBadge.tsx`/`SlaFirstResponseBadge.tsx`/`SlaProgressBar.tsx` (all under `components/sla/`) exist **only** in the RBAC-embedded copy (`unified-frontend/src/ticket-workspace/components/sla/`) — this standalone `frontend/` app has no `components/sla/` directory at all, consistent with the dual-copy rule at the top of this file (only the embedded copy has been iterated on for this feature). Don't assume feature parity without checking.

## Assignable Agents (Assigned-To picker)

`app/services/assignment_service.py`'s `AssignmentService` resolves who the current user is allowed to assign a ticket to when promoting a pending email (`GET /agents/assignable`, `listAssignableAgents` on the frontend) — a role-scoped, hierarchy-aware picker rather than the older, simpler `GET /agents?category=<name>` (`listAgents`, still used elsewhere, e.g. the plain Transfer-Agent picker, not replaced everywhere). Role rules: **Account Manager** sees their own reporting Team Leads and Staff (`manager_id` match) plus themselves; **Team Lead** sees their own reporting Staff (`teamlead_id` match) plus themselves; **Site Lead**/**Super Admin** see every active Account Manager/Team Lead/Staff, unrestricted. `get_assignable_groups` is the read-only picker data; `resolve_target` is the separate write-path guard `InboxTicketService.create_ticket_from_interaction` calls so a crafted `agent_id` in the request body can never assign outside the caller's own resolved hierarchy — never trust a client-submitted `agent_id` as-is for this reason, always route it through `resolve_target` first.

## Frontend architecture

Layering: `pages/*` (routed screens) → `components/{ticket,inbox,layout,common}/*` → `context/*` (global state) → `api/*` (one file per backend resource, thin axios wrappers) → `lib/*` (pure formatting/derivation helpers). `@/` resolves to `frontend/src` (`@tw/` to the same tree inside the RBAC-embedded copy) — use it instead of relative `../../` imports. Remember the dual-copy rule from the top of this file.

**Auth**: `context/AuthContext.tsx` + `components/layout/AuthGuard.tsx` + `pages/LoginPage.tsx` are the real login flow, added after this app moved off the old `agent_name` model — there is no agent-name switcher anymore, `currentUser` (with `.role`) is resolved from RBAC's `/me`. Login/refresh/`me` calls go through a **separate** axios instance, `api/authClient.ts` (base URL `VITE_RBAC_API_BASE_URL`), pointed at RBAC's API — not this service's own backend — since RBAC is the sole token issuer. `api/client.ts` (this service's own API) attaches the same `localStorage` access token as an `Authorization: Bearer` header via a request interceptor and, on a 401, calls RBAC's refresh endpoint (through `authClient`) via a single deduped `refreshPromise` before retrying — the pattern is ported from the shell's own frontend (`unified-frontend/src/lib/api.ts`) so both apps handle token refresh identically.

**Every API call goes through `useApiAction`** (`hooks/useApiAction.ts`), which wraps an `api/*` function with loading state and automatic toast feedback (`context/ToastContext.tsx`) — including guarding against stale responses when the logged-in user or route changes mid-request. Don't hand-roll try/catch + loading state around a new API call; wrap it with `useApiAction` and it gets error toasts for free. Backend error `detail` strings are normalized into a single readable `Error` message by an axios response interceptor in `api/client.ts`.

**`WorkflowContext`** (`context/WorkflowContext.tsx`) no longer holds an identity string — the current agent comes from `AuthContext` (the real, authenticated RBAC user) instead. `WorkflowContext` now only holds `agents: AgentSummary[]` (the directory used by the Transfer-Agent picker — a different concept from "who am I") plus the currently-open ticket, its timeline, and the selected inbox email/thread, shared across `TicketDetailPage`/`InboxPage` and their child panels (`TicketHeader`, `TicketTimeline`, `TicketDetails`, `TicketActions`, `TicketAuditLog`) instead of prop-drilling.

**Role-gated nav/UI**: `useMailInbox.ts`'s `SUPERVISOR_ROLES` constant decides who gets the "All Inboxes" escape-hatch view vs. the AM-scoped views. **It does not actually mirror the backend's `SUPERVISOR_ROLE_NAMES`** (`{"Team Lead", "Account Manager", "Site Lead", "Super Admin"}`) — the frontend constant is only `["Site Lead", "Super Admin"]`, missing Team Lead and Account Manager. This is a known, pre-existing drift (not yet fixed): the backend's `scope=all` already works correctly for a Team Lead/Account Manager who somehow requests it, they just have no UI control to trigger it. Don't assume these two sets match — re-read both before relying on either. `Sidebar.tsx`'s nav items are role-conditional via a `hideForRoles: string[]` field per item (e.g. "Create Dummy Mail" hides for `["Staff", "Team Lead"]`) — check this against `access_control.py`'s role sets before assuming a nav item's visibility matches the backend's actual gating; they've drifted from each other before. The tickets list page also has a client-side category filter dropdown (derived from whatever `ticket_type` values are already in the fetched set) — since the backend now scopes Team Lead/Staff to their own category server-side, this filter is effectively a no-op single-option dropdown for those two roles, and only meaningfully filters for Account Manager/Site Lead/Super Admin, who see multiple categories at once.

**Adding a ticket action tile** (see `components/ticket/TicketActions.tsx`): each action is an `ActionTile` that opens one of a small set of modals, calls its `api/interaction.ts` function through `useApiAction`, then calls the parent's `onActionComplete` to refetch the ticket + timeline + audit log. Any new interaction/audit-event type needs a matching entry in `lib/interactionMeta.ts` (timeline icon/label/summary) and `lib/auditLogMeta.ts` (audit panel icon/label) or it renders with a generic fallback. The "Transfer Agent" tile is a good template for a role-gated + category-scoped action: it's hidden entirely for Staff (`isStaff` check against `useAuthContext().currentUser?.role`, not just left to the backend's 403), fetches its agent picker via `listAgents(activeTicket.ticket_type)` — scoped to the ticket's own category, refetched whenever `activeTicket` changes — and relabels itself ("Assign to Staff" vs "Transfer Agent", based on `activeTicket.agent_id == null`) since the same action works on both an unclaimed shared-pool ticket and an already-assigned one; there's no separate "claim-then-transfer" step required. **Related Tickets is a lighter-weight variant of the same idea, inline in `components/ticket/TicketDetails.tsx` instead of a tile+modal** (a single-field "link a ticket" action doesn't need a whole modal): a `<select>` populated from `listTickets()` (filtered client-side to exclude the current ticket and anything already related), `addRelatedTicket`/`removeRelatedTicket` calls through `useApiAction`, then an `onRelatedChanged` prop (wired to `TicketDetailPage`'s existing `refreshAll`, the same refetch `TicketActions.onActionComplete` triggers) so the panel reflects the link/unlink immediately.

**Fine-grained permission gating reads `currentUser.permissions` directly, distinct from the role-name `isStaff`-style hides above**: Change Priority and (for Staff specifically) Transfer Agent compute `canChangePriority`/`canTransfer` from `(currentUser?.permissions ?? []).includes("ticket:...")` and **disable-in-place** (`disabled={isTicketClosed || !canChangePriority}`, with a `title` tooltip explaining why) rather than hiding the tile — the same treatment `isTicketClosed` already gets, so the action stays discoverable even when it's currently unavailable. Don't generalize this into hiding the tile entirely; that's the `isStaff`-style pattern reserved for structural, not per-capability, restrictions. **`EditAccessPanel.tsx`** is a third "inline in the details column" pattern alongside Related Tickets, but for a multi-step request/approve/reject workflow rather than a single mutating action — see `CLAUDE.md`'s "Edit access requests" section for the backend it drives.

**The Mail page — this section now describes only this standalone `frontend/` app, not the embedded copy anymore.** A later redesign (two-panel layout with no third/fourth column, a restructured Message Details view, a real auto-saving Draft composer) was implemented **only** in `unified-frontend/src/ticket-workspace/` — see that repo's `CLAUDE.md`'s "Mail v2" section for the full detail. It replaced the embedded copy's `components/inbox/{MailSidebar,MailDetailsPanel,EmailDetails,AgentInbox}.tsx` (described below) with `components/mail/{MailSidebar,MessageList,MessageDetailsView,ComposeView,ReplyComposer,...}.tsx`; this standalone app's own files below were **not** touched to match. Porting that redesign back here hasn't been done — treat the two Mail pages as genuinely different UIs sharing only the backend API until that's reconciled.

`pages/InboxPage.tsx` (nav label "Mail" — renamed from "Inbox", see `lib/i18n/translations.ts`'s `nav.inbox` key in the RBAC-embedded copy) is built from `hooks/useMailInbox.ts`, a single hook owning every piece of Mail state — called once in `InboxPage`, spread into presentational children as props (never called a second time inside a child, which would mean two independent fetches and two drifting `activeView` states). Its shape:
- **Views** (`MailViewKey`): the backend's own `view=` values (`pending`/`replied`/`ticketed`/`archived`/`snoozed`/`all`) plus three client-derived-or-separately-fetched ones — `unassigned`/`mine` (sliced from the already-fetched `pending` set, no extra request), and `sent`/`drafts` (their own `GET /inbox/sent`/`GET /inbox/drafts` fetches, adapted into `InboxItem` shape via `sentItemToInboxItem`/`draftItemToInboxItem` so the one message-list component (`AgentInbox.tsx`) can render every view without a parallel row component — clicking a Sent/Draft row opens `root_interaction_id ?? interaction_id`, since `GET /inbox/{id}` expects a root-shaped id).
- **Folders** are an orthogonal filter dimension, not a `MailViewKey` — `activeFolder`/`setActiveFolder` are separate state from `activeView`/`setActiveView`, and selecting one clears the other (see the two setters' mutual-exclusion logic). `MailSidebar.tsx` renders the Folders section (list + inline create/delete) alongside the view list.
- **Mail Details** (`components/inbox/MailDetailsPanel.tsx`) owns the tags editor (pill input, full-replace on every add/remove — matches the backend's full-replace semantics, no per-tag diffing), the folder `<select>`, and the snooze control (preset buttons + Unsnooze, hidden once a thread is ticketed — mirrors the backend's own ticket_id guard).
- **The reply composer** (`components/inbox/EmailDetails.tsx`) also owns the draft lifecycle: `useEffect`-prefills `message` from `selectedEmail.draft_message` whenever the selected thread changes (never carries over typed text between threads), "Save draft" calls `onSaveDraft`, and "Send" branches on `hasDraft` — if a draft exists it re-saves the current text first (so an edit-then-send in one motion never loses text or leaves a stale draft row) then calls `onSendDraft`; otherwise it's a plain `replyToInteraction` call, unchanged from before Drafts existed.
- Every piece above (types, `api/inbox.ts`/`api/mailFolder.ts`, `useMailInbox.ts`, `MailSidebar.tsx`, `MailDetailsPanel.tsx`, `EmailDetails.tsx`, `AgentInbox.tsx`) is mirrored into the RBAC-embedded copy with the same `@/` → `@tw/` import swap as everything else in this file's dual-copy rule.

**Dashboard stats** (`pages/Dashboard.tsx`) are derived client-side from `listTickets` + `getInbox` fetched once in a `useEffect` — there is no dedicated stats endpoint. New KPI cards should generally be a `filter`/`derive` over the already-fetched `tickets` array rather than a new network call.

**Theme**: light/dark is CSS-variable-driven (`src/index.css`) + Tailwind `darkMode: "class"`, toggled in `Topbar` and persisted to `localStorage`; an inline script in `index.html` applies the saved theme before React mounts to avoid a flash of the wrong theme. Inside the RBAC-embedded copy, `.tm-scope` (RBAC's `globals.css`) remaps this multi-hue palette onto RBAC's own monochrome one instead.
