## Repo layout

This is a monorepo formed by merging three previously-independent git repos via `git subtree` (full history preserved — `git log --follow` works across the merge point), **plus a later backend unification and a frontend consolidation** (see the two callouts below — read them before trusting any `rbac-service/backend/...`, `rbac-service/frontend/...`, or `ticketing-service/backend/...` path anywhere else in this file or either sub-`CLAUDE.md`; `rbac-service/` no longer exists as a directory at all):

- `unified-frontend/` — the **shell application** (formerly `rbac-service/frontend` — the directory was renamed to a monorepo-root sibling once it was confirmed to already be a strict superset of `ticketing-service/frontend`; see "The ticket workspace exists in two places" below). Owns authentication, session, roles/permissions/users, and role-based routing/navigation for the whole product, plus an embedded copy of the ticket workspace. Next.js 16. Has its own `CLAUDE.md` (moved here along with the frontend it describes, from the old `rbac-service/CLAUDE.md`) — read it before working in this directory. `rbac-service/` itself no longer exists — its `backend/` moved to `unified-backend/` earlier, its `frontend/` and `CLAUDE.md` moved here, and its remaining leftover docs (`docs/{API,ARCHITECTURE,DEPLOYMENT}.md`) moved into `unified-frontend/docs/` too; a stale scaffold `README.md` and an empty stray `package-lock.json` were deleted rather than carried forward. Don't expect to find an `rbac-service/` directory in this repo anymore.
- `ticketing-service/` — the **ticket management product** — a standalone, independently-runnable Vite/React frontend with its own login flow (docs/history describe a once-independent backend too; see "Backend consolidation" below for why that's no longer accurate). Its frontend is **also embedded** inside `unified-frontend` (see below). Has its own `CLAUDE.md`, though its backend-architecture sections now describe history, not the current runtime.
- `shared_models/` — the one real copy of the `User`/`Role` SQLAlchemy models the backend imports (as a local editable install, not a git-URL pip dependency). Never edit these models from a service's own repo copy — there isn't one.
- `unified-backend/` — **the actual, currently-running backend for everything** (see below). Has no `CLAUDE.md` of its own yet; this file and each service's `CLAUDE.md` cover it. okay

## Backend consolidation: one FastAPI app, not two

`rbac-service/backend/` and `ticketing-service/backend/` **no longer exist as separate running services** — they were merged into a single `unified-backend/` FastAPI app (`unified-backend/app/main.py`), discovered mid-session when `ticketing-service`'s old port-8001 process turned out to have never existed post-merge and every "network error" traced back to the frontend still pointing at it. Both services' own `CLAUDE.md` "Commands"/local-dev sections have since been corrected to point at `unified-backend/` — if you ever see one describing a standalone `backend/` directory or a separate port again, that's drift to fix, not the current state; the ticket-workspace/RBAC frontend-side documentation in both files was already accurate and untouched.

Layout: `unified-backend/app/rbac/` (former `rbac-service/backend/app/`) and `unified-backend/app/ticketing/` (former `ticketing-service/backend/app/`) are mounted in the same `FastAPI()` instance — RBAC's routers under `/api/v1` exactly as before, Ticketing's routers unprefixed exactly as before (see `app/main.py`'s router-mounting comment) — so **every existing route path is byte-identical to the two standalone services**, just served from one process on one port (`:8000` by default) instead of two. `app/notifications/` is a third, newer module living alongside them (see `unified-frontend/CLAUDE.md` if a notifications feature needs touching). Two independent Alembic histories are preserved as `alembic_rbac/` and `alembic_ticketing/` (own `versions/` dirs, own `include_object` filtering) — `unified-backend/scripts/rbac_seed/` and `scripts/ticketing_seed/` are the corresponding seed scripts, run via `alembic -c alembic_rbac/alembic.ini upgrade head` / `alembic -c alembic_ticketing/alembic.ini upgrade head` (order matters only against a genuinely empty DB, since ticketing's tables FK into rbac's `users`). One physical Postgres database (Neon) throughout, same as before the merge.

`unified-backend/.env` is the one env file both halves read from — `JWT_SECRET_KEY`/`JWT_ALGORITHM` no longer need to match across two files since there's only one now, but the RBAC-issues/Ticketing-verifies token *architecture* described below is otherwise unchanged (it's just enforced within one process instead of across two).

## Cross-service identity: RBAC issues, Ticketing verifies

This section describes a **conceptual** boundary that survived the process merge above — read "service" below as "module" (`app.rbac` vs. `app.ticketing`), not "separate deployable."

RBAC's own code (`app.rbac.services.auth_service`) is the **sole issuer** of JWT access/refresh tokens (HS256). Ticketing's own code (`app.ticketing`) is a **verify-only consumer** — it decodes and validates tokens against the same `JWT_SECRET_KEY` (now literally the same `settings.jwt_secret_key`, read once by the one process); it has no login/signup/refresh endpoint of its own and no `create_token`-shaped function anywhere in its code. Before the merge this required byte-identical `JWT_SECRET_KEY` values across two `.env` files; now there is only one `.env` to get right. See `ticketing-service/CLAUDE.md`'s "Access control" section for the exact dependency chain (`get_current_user`/`get_current_agent`, `AGENT_ROLE_NAMES`/`SUPERVISOR_ROLE_NAMES`).

**The access token also carries a `permissions` claim, plus a separate `scoped_permissions` claim.** `app.rbac`'s `create_access_token` embeds the caller's full effective permission list (role defaults ∪ active *unscoped* personal overrides, computed by `PermissionResolverService`) at login/refresh time as `permissions`, and any *ticket-scoped* overrides (e.g. `ticket:editother_ticket` granted for one specific ticket only — see `unified-frontend/CLAUDE.md`'s "Permission requests" section) separately as `scoped_permissions: dict[str, list[str]]` (permission name → ticket ids). `app.ticketing`'s `dependencies/auth.py` threads both onto the resolved `User` as transient, non-persisted attributes (same pattern as `TicketService._attach_names`) and reads them via `access_control.has_permission`/`has_permission_for_ticket`/`ensure_has_permission` — a decode-only check, never a fresh call back into `app.rbac`. This is how fine-grained (not just role-name) authorization crosses the module boundary — see `unified-frontend/CLAUDE.md`'s "Per-user permission overrides" section for how both claims are computed and granted, and `ticketing-service/CLAUDE.md`'s "Permission-based enforcement" section for how they're consumed. A stale or absent claim degrades to an empty list/dict rather than crashing, so a token issued before either claim existed still decodes safely — and granting or revoking a permission doesn't affect a token already issued, only the next login/refresh.

**`get_current_user` no longer re-resolves the user from Postgres on every request — only on a cache miss.** The access token also now carries `name`/`role_id`/`category_id`/`category` (the `CategoryName` string value) and `permission_version` (all optional/backward-compatible — a token minted before these claims existed just always takes the DB path below, degrading safely like `permissions`/`scoped_permissions` already did). `app.ticketing`'s `get_current_user` (`app/dependencies/auth.py`) checks an in-memory, per-process, TTL-based cache (`app/core/rbac_cache.py`, default TTL 30s, no Redis) keyed on `(user_id, permission_version)`: on a hit, it reconstructs a transient, never-persisted `User`/`Role`/`Category` straight from the JWT claims — zero DB round trips. On a miss (first request for that token, TTL expired, or a `permission_version` never seen before), it does fall back to the original `UserRepository.get_by_id` (`joinedload` of `User`+`Role`+`Category`, one round trip), and if the DB's live `permission_version` doesn't match the token's claimed one, the request is rejected outright (401, "session outdated") rather than trusting stale role/category/`is_active` for the rest of the token's natural TTL. `permission_version` (a plain integer column on `shared_models.models.User`) is bumped — never scanned-for in the cache, which is what keeps invalidation O(1) — by `app/rbac/services/user_service.py` (role/category/manager/teamlead reassignment, activate/deactivate), `permission_override_service.py` (grant/revoke, on the target user), and `role_permission_service.py` (a role's own permission set changing — one bulk `UPDATE users ... WHERE role_id = ...`, not a per-user loop, since that affects everyone holding the role). Net effect: a normal authenticated read request now costs one Postgres round trip (the business query) instead of two, with RBAC changes bounded to at most one cache-TTL window of staleness per process rather than propagating instantly — an explicit, deliberate tradeoff, not an oversight.

**A frontend request-duplication bug (see `unified-frontend/CLAUDE.md`'s "Request-duplication audit" section) exposed two related gaps here, both fixed in the same pass.** `unified-backend/app/database/session.py`'s connection pool (`pool_size=10, max_overflow=20`, 30 total, no explicit `pool_timeout` — SQLAlchemy's own default is 30s) was easily oversubscribed once that frontend bug put 200-300 concurrent requests in flight at once, and requests queued for a free connection close to that 30s timeout — the actual explanation for reported "26-48 second" responses on endpoints too cheap to genuinely take that long (a 3-row `GET /sla/policies` chief among them). Raised to `pool_size=20, max_overflow=30, pool_timeout=10` (fail fast with a clear error instead of hanging near the old 30s default) — this is headroom against a future burst, not the primary fix, since the frontend bug that caused this particular burst is already fixed; don't raise it further without checking Neon's own connection ceiling on the compute backing the pooled endpoint in `DATABASE_URL` (its `-pooler` hostname). Separately, `rbac_cache.py`'s TTL cache had no protection against a stampede on a **miss**: `is_valid`/`mark_valid` only cover an already-resolved result, so N concurrent requests for the same `(user_id, permission_version)` arriving before any of them completed would each independently miss and each hit Postgres via `UserRepository.get_by_id`. `resolution_lock(user_id)` (same file) now serializes concurrent misses per user: the first caller through actually resolves and calls `mark_valid`; every other concurrent caller for that user_id waits on the lock, then re-checks the cache (now warm) and takes the normal cache-hit path instead of also querying Postgres. Locks are created and removed on demand (a plain dict + `asyncio.Lock`, no persistence), bounded to "users with a resolution currently in flight," not retained forever for every user ever seen.

## The ticket workspace exists in two places

`ticketing-service/frontend` is a real, independently-runnable Vite/React app with its own login flow. Its entire page tree was **also copied** (not just linked) into `unified-frontend/src/ticket-workspace/` (formerly `rbac-service/frontend/src/ticket-workspace/`) and mounted inside the shell's Next.js app via `react-router-dom`, so that Staff/Team Lead/Manager get one seamless product after logging in, instead of being bounced to a separately-hosted app. See `unified-frontend/CLAUDE.md`'s "Ticket workspace embedding" section for the exact mounting mechanism, routing rules (which roles land where), and the design-token unification that makes the embedded copy visually match the shell's own pages.

**These two copies do not stay in sync automatically.** A change to a ticket page, component, or API wrapper generally needs to be made in both `ticketing-service/frontend/src/...` and `unified-frontend/src/ticket-workspace/...` (same relative paths; the embedded copy imports via the `@tw/*` alias instead of `@/*`) if it should be visible in both the standalone app and the embedded experience. Check which one(s) are actually reachable/deployed before assuming a one-sided change is sufficient.

Both copies now call the **same** `unified-backend` process (see "Backend consolidation" above), just via different base-URL env vars — `ticketing-service/frontend`'s own `VITE_API_BASE_URL` and the embedded copy's `NEXT_PUBLIC_TICKETING_API_URL` should both point at the unified backend's root (no `/api/v1`) now that it's one port instead of two.

**The gap between the two copies has widened, not narrowed.** A Mail-page rebuild (two-panel layout, redesigned Message Details, a real auto-saving Draft feature — see `unified-frontend/CLAUDE.md`'s "Mail v2" section) was implemented **only** in the embedded copy, since the standalone `ticketing-service/frontend` app wasn't the one being tested/iterated on. The standalone app's own Mail/Inbox page still reflects the older design documented in `ticketing-service/CLAUDE.md`. Don't assume the two `CLAUDE.md`s' Mail-related sections describe the same UI anymore — they don't, and reconciling them (porting the newer design back to the standalone app) hasn't been done.

**As of the frontend consolidation, `rbac-service/frontend` no longer exists as a separate directory** — it was renamed to the monorepo-root `unified-frontend/` (mirroring `unified-backend/`'s naming), since it was already a strict superset of `ticketing-service/frontend` (same routes, plus the Mail v2 work above that the standalone app never got). `rbac-service/` itself now holds only historical docs (its old `CLAUDE.md` moved to `unified-frontend/CLAUDE.md` along with the code it describes). `ticketing-service/frontend` is unaffected and still runs standalone — this consolidation only retired the *duplicate* embedded copy's old home, not the standalone app itself (that's a separate, later decommissioning step).

## Local development

Two processes for full end-to-end testing (the unified backend, and the unified frontend that embeds the ticket workspace — the standalone Ticketing frontend is a third, optional one only needed if testing that app directly rather than the embedded copy):

```bash
cd unified-backend && bash scripts/start.sh     # :8000 — runs both Alembic chains, then uvicorn
cd unified-frontend && npm run dev              # :3000, embeds the ticket workspace
```

`scripts/start.sh` assumes `unified-backend/.env` already exists (`DATABASE_URL`, `ALEMBIC_DATABASE_URL`, `JWT_SECRET_KEY`/`JWT_ALGORITHM`, storage vars — see `unified-backend/app/core/config.py`'s `Settings` for the full field list). If you'd rather run the pieces individually: `alembic -c alembic_rbac/alembic.ini upgrade head`, `alembic -c alembic_ticketing/alembic.ini upgrade head`, then `uvicorn app.main:app --reload --port 8000`, all from `unified-backend/`. `Settings` is cached via `@lru_cache` — editing `.env` while the backend is already running has no effect until it's restarted (`--reload` only reacts to Python file changes, not `.env` edits, and has been unreliable enough on Windows in this repo that a full process kill + restart is the trustworthy fix when a change "isn't taking effect").

**Both frontends need their ticketing-API base URL pointed at the unified backend's root (no `/api/v1`), not a separate `:8001`.** `unified-frontend`'s embedded ticket workspace defaults `NEXT_PUBLIC_TICKETING_API_URL` to the old standalone-ticketing-service port (`http://localhost:8001`) if unset — since that port no longer has anything listening on it post-merge, this silently network-errors every ticketing-domain request while RBAC-native requests keep working. Set `NEXT_PUBLIC_TICKETING_API_URL=http://localhost:8000` in `unified-frontend/.env.local` (restart `npm run dev` after — `NEXT_PUBLIC_*` vars are baked in at server start). See `unified-frontend/CLAUDE.md`'s matching Known Issues entry.

See each frontend's own `CLAUDE.md` and `ticketing-service`'s `run` skill for the full prerequisite/verification checklist (the skill still describes the old two-backend shape in places — mentally translate `backend/` paths to `unified-backend/app/ticketing/` per "Backend consolidation" above).

## Deployment

Render.com, via the single root-level `render.yaml` (not a per-service `render.yaml` — there is only the one, at the repo root): one `unified-backend` Web Service (rootDir `unified-backend`), one `rbac-frontend` Web Service (rootDir now `unified-frontend` — the Render service name itself was deliberately left unchanged to avoid creating a new URL/hostname; only its `rootDir` moved), and one `ticketing-frontend` static site — see `DEPLOYMENT.md` at the repo root for the full runbook (Neon setup, the CORS/API-URL circular-dependency first-deploy sequence, seeding). Rotating `JWT_SECRET_KEY` in production is a real, disruptive step — every currently-issued token becomes invalid immediately, forcing a global logout. Schedule it deliberately; don't treat it as a routine env var change.

**The SLA sweep now runs in-process via APScheduler — GitHub Actions is a manual fallback only, not the trigger.** An earlier revision of this file described the sweep as GitHub-Actions-triggered (`render.yaml` never having a usable `type: cron` service, since Render Cron Jobs require a paid plan and this Blueprint is entirely free-tier); that's since been superseded. `unified-backend/app/core/sla_scheduler.py` (new) wires an `AsyncIOScheduler` singleton into `app/main.py`'s existing `lifespan` (`start_scheduler()`/`shutdown_scheduler()`), firing `SLASweepService.run_sweep` on an interval controlled by `SLA_SWEEP_INTERVAL_MINUTES` (default 1) directly inside the same running backend process — no external caller and no shared secret to keep in sync for the routine case. `max_instances=1`/`coalesce=True`/`replace_existing=True` guard against overlapping ticks if one run ever takes longer than the interval. `.github/workflows/sla-sweep.yml`'s `schedule:` trigger was removed accordingly — its `workflow_dispatch: {}` manual trigger was kept as an emergency/manual fallback only, and `POST /internal/sla/sweep` is still shared-secret-protected and reachable for that purpose (`SLA_SWEEP_URL`/`SLA_SWEEP_SHARED_SECRET` GitHub repo secrets, still kept byte-identical to `unified-backend`'s own `SLA_SWEEP_SHARED_SECRET` env var for when it's used). `scripts/sla_sweep_local_loop.py` (a local-dev workaround for the old gap) was deleted as redundant, and `render.yaml` gained the matching `SLA_SWEEP_INTERVAL_MINUTES` env var. Don't reintroduce a second external scheduler for this — one in-process trigger per tick is an invariant `EscalationService.evaluate_overdue`/`EscalationHandlingSlaService.evaluate_breaches` already depend on running exactly once per sweep (see below), not twice from two competing triggers.

## SLA & Escalation

A substantial feature living entirely in `unified-backend/app/ticketing/` (never ported to the standalone `ticketing-service/frontend` — see that repo's own CLAUDE.md and the "ticket workspace exists in two places" pattern above; the SLA UI components exist only in `unified-frontend/src/ticket-workspace/components/sla/`).

Two independent per-ticket clocks, not one: **First Response SLA** (`FirstResponseSLA` — starts when a ticket's founding interaction lands, completes on the first agent reply) and **Resolution SLA** (`ResolutionSLA` — starts at ticket creation, pauses while `current_status == WAITING_FOR_CLIENT`, reshifts its `due_at` on a priority change, completes only when a supervisor closes the ticket — entering `RESOLVED` does **not** complete it, only `CLOSED` does). Target minutes per priority live in `SLAPolicy`, a lookup table, not hardcoded — and the four thresholds themselves aren't fixed constants either: `SLAPolicy.warning_1_percentage`/`warning_2_percentage` (plus `handling_sla_percentage`, see the `EscalationHandlingSLA` paragraph below) are per-priority-tier columns feeding directly into `SLASweepService`'s per-tick `thresholds_reached(fraction, half_elapsed=policy.warning_1_percentage/100, at_risk=policy.warning_2_percentage/100)` call — editable live via the Super-Admin-only SLA Timing Matrix page (`unified-frontend/CLAUDE.md`'s matching section), not a code change. A periodic sweep (`SLASweepService.run_sweep`, triggered by the in-process APScheduler described under "Deployment" above, and still separately reachable via `POST /internal/sla/sweep`, shared-secret-protected — `settings.sla_sweep_shared_secret`, no default, fails fast at boot if unset) evaluates every active clock's elapsed fraction against four thresholds (`HALF_ELAPSED`/`AT_RISK`/`BREACHED`/`ESCALATED`), records each crossed `(clock_type, clock_id, threshold)` triple in an idempotency ledger (`SLABreachNotificationRepository.try_record_many`, one batched `INSERT ... ON CONFLICT DO NOTHING ... RETURNING`) so a threshold only ever notifies once, then fires both an in-app notification and a real outbound email (falls back to logging-only until `smtp_host` is configured).

**First Response SLA notifications now open the specific message, not just the inbox, and carry real content instead of one generic template.** `sla_breach_notifier.py`'s `notify_first_response_threshold` used to hardcode `link="/inbox"` and an identical title/message for every threshold crossing regardless of which mail triggered it. `link` is now `f"/inbox?interaction_id={clock.interaction_id}"` (mirroring the Resolution SLA notification path's existing `f"/tickets/{ticket.ticket_id}"` shape), and a new `_first_response_notification_copy()` helper builds a subject/client/body-snippet-specific title and message (snippet capped at `BODY_SNIPPET_MAX_CHARS = 200`) from the triggering `Interaction` row — fetched via a batch prefetch (`fr_interaction_ids`/`fr_interactions_by_id` in `sla_sweep_service.py`, mirroring the pre-existing `fr_clients_by_id` prefetch pattern: one query per sweep tick for every clock crossing a threshold, never a per-clock fetch) for the periodic sweep path, and a single direct fetch for the one-off `complete_first_response_clock` completion path. See `unified-frontend/CLAUDE.md`'s matching section for the `InboxPage.tsx`/`RouterSync` mechanics that make clicking the notification actually open that message.

**A separate internal escalation workflow** (`TicketEscalation`, `EscalationService`) rides on top of — but is architecturally forbidden from touching — the Resolution SLA clock's own `started_at`/`due_at`/`status` columns (this is the single invariant the feature's own pytest suite exists to guard, per-test). An escalation auto-creates at `TEAM_LEAD` level on a ticket's first `BREACHED`/`ESCALATED` crossing (`auto_escalate_if_needed`, a no-op if one's already active), can also be triggered manually (`manual_escalate`, requires `ticket:escalate`), and auto-advances through `TEAM_LEAD → MANAGER → SITE_LEAD` if its `ack_due_at` lapses with no acknowledgment (`evaluate_overdue`, run at the end of the same sweep tick — not a second scheduler). `SLA_ESCALATED` (the Resolution SLA's own notification-ladder tier) and `ESCALATION_CREATED`/`_ACKNOWLEDGED`/`_ADVANCED`/`_CLOSED` (the internal ownership-chain events) are deliberately distinct `AuditEventType` values — don't conflate them.

**Three new Postgres-native enums** landed with this feature: `SLAClockStatus` (`sla_clock_status_enum`), `EscalationLevel` (`ticket_escalation_level_enum`), `EscalationStatus` (`ticket_escalation_status_enum`) — see `ticketing-service/CLAUDE.md`'s "add-postgres-enum-value" skill reference before adding a member to any of these or to `AuditEventType`'s new `SLA_*`/`ESCALATION_*` values.

**`SLA_PAUSED`/`SLA_RESUMED` are real audit events now, not silent state changes.** These were `SLA_MANUALLY_PAUSED`/`SLA_MANUALLY_RESUMED` until this session, and only the manual-override endpoints (`SLAService.manual_pause`/`manual_resume`) ever wrote one — the automatic pause/resume driven by `InteractionService.change_status` whenever a ticket enters/leaves `WAITING_FOR_CLIENT` (calling `SLAService.pause_resolution_clock`/`resume_resolution_clock`) logged nothing at all, an audit-trail gap since the clock itself was still pausing/resuming correctly. Both enum members were renamed (`ALTER TYPE ... RENAME VALUE`, `alembic_ticketing/versions/a2c4e6f8b0d3_rename_sla_manually_paused_resumed.py`, down_revision `b7f1d3e5a9c2`), and `change_status` now logs `SLA_PAUSED` unconditionally on entering `WAITING_FOR_CLIENT` and `SLA_RESUMED` specifically when the new status is `IN_PROGRESS` or `RESOLVED` (deliberately not `CLOSED` — that transition is already covered by `TICKET_RESOLVED`/`ESCALATION_CLOSED` semantics). The manual-override path still writes under these same (renamed) events rather than a separate pair — distinguished only by a `"trigger": "manual_override"` key in `new_values`, so one event type now covers both triggers and both are visible in a ticket's Audit Log tab. (`SlaTimeline.tsx`, a separate ticket-detail card that used to be the only other place these events surfaced, was removed entirely as redundant with the Audit Log tab — see `unified-frontend/CLAUDE.md`'s matching section.)

**A second internal clock, `EscalationHandlingSLA`, was added on top of the escalation workflow above** (`app/ticketing/models/escalation_handling_sla.py`, own table `escalation_handling_slas`, own repository/service) — it measures time-to-actually-resolve once the current escalation owner has acknowledged (or been assigned) the ticket, entirely independent of both `ResolutionSLA` and `TicketEscalation`'s own columns; nothing in this class ever writes to either. Its target is always `round(original_resolution_target_minutes * 60 * 0.25)` — 25% of the *original* Resolution SLA's configured target duration, computed once in `EscalationHandlingSlaService.compute_escalation_handling_target_seconds` (the one and only place this formula lives) — never derived from remaining/overdue time. `EscalationService.acknowledge` and the new `acknowledge_via_assignment` (called from `InteractionService.transfer_agent` when a supervisor assigns an escalated ticket to staff — assignment before acknowledgment counts as acceptance) both start this clock via `EscalationHandlingSlaService.start_if_not_started`, which is idempotent by design: an existing row for that `escalation_id` is returned unchanged rather than re-created, so acknowledging twice or reassigning later never restarts the timer. `SLASweepService.run_sweep` now also calls `EscalationHandlingSlaService.evaluate_breaches` at the end of its tick (same sweep, not a second scheduler) — a breach marks `breached_at`, notifies the current owner, and calls `EscalationService.advance_for_handling_sla_breach` to move the escalation to the next configured level, reusing the existing `ESCALATION_ADVANCED` event (tagged `reason: "escalation_handling_sla_breach"` in its `new_values` JSON to distinguish it from an ack-window-based advance) rather than adding a new `AuditEventType`. Migration `d33a0758e3c4_add_escalation_handling_slas.py` adds only this one table — an autogenerate diff also proposed dropping several unrelated indexes/columns (`ix_tickets_updated_at`, `ix_tickets_pool_view`, `ix_tickets_title_trgm`, some `interactions` indexes, `interactions.snoozed_until`) that are real, deliberately-added prior work simply not declared in the current SQLAlchemy models — that drift was left alone and stripped out of this migration rather than silently applied as a side effect of an unrelated feature.

**The ticket-list page gained a role-gated "Escalated" tab** (`view=escalated` on `GET /tickets`, plus a matching `escalated` count on `GET /tickets/view-counts`) — visible only to `ESCALATION_TAB_ROLE_NAMES` (`access_control.py`: Team Lead, Account Manager, Site Lead, Super Admin, mirroring `SUPERVISOR_ROLE_NAMES`), and filtered by the exact same AM-owned-clients/TL-own-category visibility scoping every other view already applies — an Account Manager only ever sees escalations on tickets they own, never a company-wide list, with no separate authorization path to maintain (verified live: a Team Lead saw 4 escalated tickets, the Account Manager scoped to those clients saw 2 of the same 4, Site Lead saw all 4). `TicketRepository.list_visible_page`'s `view == "mine"` branch also gained a fixed, escalation-aware `ORDER BY` (actively-escalated first, then HIGH priority, then nearest Resolution SLA deadline, then the caller's own chosen sort, then `ticket_id` as a deterministic tie-breaker) computed entirely in SQL — an escalated ticket assigned to staff floats to the top of their My Tickets tab with no client-side sorting involved. `TicketResponse`/`TicketListItemResponse` carry `is_escalated`/`escalation_level`/`escalation_status`/`escalation_ack_due_at` (LEFT JOIN-sourced, no per-row lookup) purely for display — a ticket's real `current_priority` is never overwritten by escalation state; the frontend renders a separate "CRITICAL" badge instead (see `unified-frontend/CLAUDE.md`'s matching section). No new acknowledge endpoint was added — `POST /tickets/{id}/escalation/acknowledge` is unchanged; acknowledging via assignment reuses `InteractionService.transfer_agent`/`POST /tickets/{id}/transfer` rather than a second assignment mechanism, which gained a category-scope validation step specifically for the case where the target ticket has an active escalation.

**The Acknowledge & Assign candidate picker (referenced above, and the same one the CRITICAL-priority section below points to) is now role-scoped, not a flat Staff-only list.** `EscalationService.get_acknowledge_candidates` (backing a new `GET /tickets/{ticket_id}/escalation/acknowledge-candidates` route in `api/sla.py`) replaces the earlier flat picker with grouped candidates that depend on who's acknowledging: a Site Lead/Super Admin gets the ticket's category's Team Lead(s) plus the client's own Account Manager; an Account Manager gets their own category-matched, reporting Team Lead(s); a Team Lead still gets their category's Staff, unchanged from before. Reuses the existing `AssignableUserSummary`/`AssignableGroup`/`AssignableAgentsResponse` schemas (the same shapes `AssignmentService` already returns elsewhere) rather than inventing new ones — see `unified-frontend/CLAUDE.md`'s matching section for the `<optgroup>`-grouped rendering this backs.

**Unclaimed, escalated tickets are excluded from the Open Pool, and are treated like a claimed-and-escalated ticket for visibility purposes — reachable only via the Escalated tab, with no auto-assignment.** `TicketRepository.list_visible_page`'s `pool` branch and `list_all`'s equivalent `pool` branch both gained a condition excluding any ticket with an active `TicketEscalation` row (`TicketEscalation.escalation_id.is_(None)` / `not_(self._escalated_exists_condition())` respectively). Before this, an unclaimed ticket that breached and auto-escalated stayed visible — and claimable — in Open Pool despite already having an active escalation chain, which undercut the point of escalating it in the first place. By deliberate design this does **not** touch `agent_id` — it stays null, exactly as it already does for a ticket that was claimed before it escalated; the ticket only becomes reachable again through the Escalated tab and the normal Acknowledge & Assign flow above. Priority already becomes `CRITICAL` automatically on any escalation regardless of claim state (see the CRITICAL-priority section below) — no separate logic was needed for that part.

**A related, separate test fragility was found while adding coverage for the escalation-handling SLA above**: `test_escalation_service.py::test_overdue_active_escalation_advances_without_touching_sla` asserts `evaluate_overdue` advances exactly 1 escalation, but that method's own query scans the *entire* `ticket_escalations` table, not just rows created within the test's own rolled-back transaction — two real leftover `ACTIVE` escalations from earlier manual/live-testing sessions ("sla testing final ticket", "servce level agrement 1", both already advanced to `SITE_LEAD` by a real sweep run) sit permanently in the shared dev database and get counted too, inflating the assertion from `1` to `3`. Confirmed via direct query this is pre-existing dev-data debris, not a regression from any code change — the test's own isolation assumption (that it's the only active escalation in the table) doesn't hold against a shared, never-reset dev database. Not fixed (would mean either scoping the test's own assertion to just its own created ticket_id, or cleaning up the two leftover rows) — know to expect this specific failure when running this test file until one of those happens.

**A parallel-track integration pass merged this SLA/escalation/performance work with an independently-developed RBAC/audit-log/user-management pass** (a `git stash pop`, not a `git merge` — no `MERGE_HEAD`, so don't go looking for one). Both sides' work was preserved, but the integration surfaced (and fixed) several real, pre-existing bugs worth knowing about if something in this area still looks off: a missing `selectinload` import in `unified-backend/app/rbac/repositories/user_repository.py` that 500'd `GET /users` and five other methods with no CORS headers on the error response (looks like a CORS failure in the browser, isn't one — see `unified-frontend/CLAUDE.md`'s Known Issues); the `AuditEventType` Python enum briefly had the same 8 SLA/escalation values declared twice; two Alembic migrations in different chains (`alembic_rbac`/`alembic_ticketing`) ended up with the identical revision id `b3d5f7a9c1e2` (harmless — each chain uses its own `version_table`, `ticket_alembic_version` for ticketing — but worth knowing before assuming revision ids are unique repo-wide); and genuinely-broken, never-executed dead code (`getTicket`/`setTicketDetail`, neither ever imported/declared) in `MessageDetailsView.tsx`, removed rather than carried forward. `unified-backend/tests/` also gained a real pytest suite during this — pure-logic tests (SLA clock math, escalation rules, sweep-auth) run fine together; DB-touching tests (`test_escalation_service.py`, `test_interaction_threading.py`, `test_get_current_user_cache.py`) each pass individually but **hang if more than one runs in the same pytest process** — a pre-existing pytest-asyncio issue (a module-level async engine's connection pool gets bound to whichever event loop existed when it was first used, and pytest-asyncio's default per-test event loop scope means a second test's fresh loop can never complete operations queued against the first loop's transport), not something introduced by the integration. Not yet fixed (would mean adding `asyncio_default_fixture_loop_scope` config or a per-test engine) — know to run DB-touching test files one test at a time until it is.

## CRITICAL priority: a real, escalation-only, permanent priority tier

`TicketPriority` gained a fourth member, `CRITICAL` (`app/ticketing/enums/ticket_enums.py`), widened into the Postgres-native `ticket_priority_enum` type via a standalone migration (`alembic_ticketing/versions/9c4e6a8b1d3f_add_critical_to_ticket_priority_enum.py`, `ALTER TYPE ... ADD VALUE IF NOT EXISTS`, per the repo's own `add-postgres-enum-value` skill) plus a new `SLAPolicy` row (`b7f1d3e5a9c2_add_critical_sla_policy_row.py`) with deliberately tighter targets than HIGH: 5 min First Response, 60 min Resolution, 10 min Escalation Ack Window, 25% Handling SLA, 50%/80% warning tiers. This is a real, filterable/sortable priority value on `Ticket.current_priority` — not the pre-existing display-only "CRITICAL" badge the ticket-list/escalated-tab UI already rendered for `is_escalated` tickets (see the "Escalation-handling SLA" section's `TicketsListPage.tsx` bullet above), which is unrelated and untouched.

**By deliberate design (confirmed with the user before building this): CRITICAL is escalation-only and permanent.** It is never a manually-selectable option on any "Change Priority" surface (backend or frontend) — the only writer is `EscalationService._bump_priority_to_critical`, called as the first line of `_create_escalation` (shared by both `manual_escalate` and `auto_escalate_if_needed`, so every escalation path gets it consistently), which is a no-op if the ticket is already CRITICAL (idempotent — re-escalating after an ack-window advance must never re-reshift the clock a second time). It never reverts — not on acknowledge, not on closing the escalation, not on resolving/closing the ticket itself. The bump reuses `SLAService.reshift_resolution_clock_for_priority_change` (the exact same method manual Change Priority already calls, via a deferred import to dodge a circular import with `sla_service.py`'s module-level import of `build_escalation_service`) so the Resolution SLA clock's own `priority` column — which is what the sweep's elapsed-fraction math actually keys off, not `Ticket.current_priority` — stays consistent with the ticket's new tier, and writes a `PRIORITY_CHANGED` audit entry attributed to `ActorRole.SYSTEM`/"Escalation workflow" rather than a real user.

**A one-time backfill was required and run** because the bump only fires going forward, from the moment this code landed — it does not retroactively touch tickets that had already escalated before then. Confirmed live: 7 of 9 then-escalated dev-database tickets were still sitting at their pre-escalation priority (mostly MEDIUM) despite having an active/acknowledged `TicketEscalation` row. Fixed with a throwaway script (not committed, not a migration — this was one-time data cleanup, not a schema change) that queried `tickets JOIN ticket_escalations` for every escalated ticket not already CRITICAL, loaded each via the ORM, and called `EscalationService._bump_priority_to_critical` directly — reusing the exact same reshift+audit-log code path a live escalation takes, rather than hand-writing a parallel UPDATE. Verified afterward: all 9 escalated tickets in the dev database show `current_priority = CRITICAL` with their `resolution_slas.priority` matching. **If CRITICAL SLA targets or the bump logic ever change again, remember there is no ongoing backfill mechanism** — any ticket that escalates from now on gets it automatically, but a bug that temporarily breaks the bump would again require a manual one-off pass like this one to catch back up, since nothing periodically reconciles `Ticket.current_priority` against `ticket_escalations` on its own.

See `unified-frontend/CLAUDE.md`'s matching section for the frontend side (the `TicketPriority` union, `Change Priority` picker exclusion, the `SlaCard.tsx` stale-priority-display bug this surfaced and its fix, and the Acknowledge & Assign feature added in the same pass).

## RBAC permission compliance audit (2026-07-14/15)

A full audit of every permission in the separately-maintained RBAC permission-matrix doc against what the running application actually enforces — driven by the doc's exact per-role Full(✓)/Override(◐) grant as ground truth, not the pre-existing (largely unenforced) code. Full per-permission findings live in `ticketing-service/CLAUDE.md`'s "Roles" and "SLA workflow" sections (backend) and `unified-frontend/CLAUDE.md`'s two matching sections (frontend) — summarized here since it touched both `app.ticketing` and `app.rbac` plus both frontend copies' shared components in one pass:

- **A critical, pre-existing security bug**: `AttachmentService.upload_attachment`'s authorization check was called without `await` — an `async` coroutine created and immediately discarded, so it silently never ran for anyone. Any authenticated agent could upload to any ticket regardless of category or client ownership.
- **Team Lead had an unconditional role-based bypass** on Close Ticket/Reopen/manual SLA override that the matrix doc marks Override-only for that role — narrowed to a new `CLOSE_REOPEN_BYPASS_ROLE_NAMES` (`{Site Lead, Super Admin}` only) with Account Manager/Team Lead/Staff all falling through to the real permission check instead.
- **Account Manager's own-clients visibility boundary was enforced for viewing a ticket but missing from nearly every mutating action** (priority/category/status change, transfer, reply, hide-interaction, audit-trail view, attachment upload/delete, SLA override) — `ensure_account_manager_owns_ticket_client` is now called alongside each of those actions' existing checks.
- **`app.rbac` had no backend permission enforcement at all before this** (every route only checked authentication, per that domain's own historical design) — Users/Roles/Permissions/Audit-Logs routes gained a new `app/rbac/services/access_control.py` (mirroring `app.ticketing`'s own `has_permission`/`ensure_has_permission`) wired into each. The highest-impact fix here: `PUT /roles/{id}/permissions` (the "Manage Permissions" editor) had no backend gate, and the frontend's own hardcoded Super-Admin-plus-Account-Manager check both missed Site Lead (holds `permission:update` by default) and over-granted Account Manager (Override-only per the doc) — both ends fixed to a real `permission:update` check.
- **`scripts/rbac_seed/seed.py` reconciled** to match the doc exactly: `ticket:close` renamed to `ticket:close_ticket`; `ticket:manage_attachments` split into `ticket:upload_attachment` (Full for everyone) + `ticket:archive_attachment` (Override-only for Team Lead/Staff); new permissions seeded for `communication:attach_to_ticket`, `ticket:view_escalated`, `ticket:acknowledge_escalation` (all three previously either unseeded or enforced by a hardcoded role allowlist with no per-user override path); several over/under-privileged default grants corrected (e.g. `ticket:view_unassigned` had actually been revoked from Staff by old seed data, contradicting the doc's Full-for-everyone grant).
- **The ticket-workspace's own Audit Log page** (`GET /tickets/audit-logs`) was the one deliberate exception to "just add the permission check": requiring `ticket:view_global_audit_log` for the whole page turned that permission's Override-only grant into "blocks the page outright" for Account Manager/Team Lead/Staff. Reworked into a scoped-by-default view (no permission required — Account Manager/Team Lead/Staff each get a real, role-accurate slice, Staff newly narrowed to just their own assigned tickets rather than their whole category) plus an opt-in centralized mode gated by that permission and reachable via a new button on the page — see `unified-frontend/CLAUDE.md`'s matching section for the button/badge/tooltip UI.
- Two items were deliberately left as architectural/scope limitations rather than "fixed": personal permission overrides are purely additive (no mechanism to grant a user *less* than their role default, so a "role allows, override revokes" scenario isn't buildable without new scope), and Related Tickets link/unlink plus the Claim Ticket button have no permission in the matrix doc covering either — both noted for whoever next revises that doc.

Every fix was live-verified against the running backend across all five roles (including a grant-then-revoke override round-trip proving the permission itself, not just the role, is what's being checked), not just type-checked or unit-tested.
