# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Enterprise RBAC (Role-Based Access Control) User Management System — a FastAPI backend + Next.js frontend monorepo. Six roles manage users, roles, and permissions across a reporting hierarchy (`manager_id` / `teamlead_id` on each user): **Super Admin** (rank 5, all permissions) > **Site Lead** (rank 4, all permissions except `ticket:system_config`/`audit:export`) > **Account Manager** (rank 3, renamed in-place from "Manager" — same `role_id`, no orphaned data) > **Team Lead** (rank 2) > **Staff** (rank 1), plus **Viewer** which sits outside the rank ladder as a client-facing role with a small fixed permission set (`user:view`, `role:view`, `permission:view`). Site Lead and Viewer are both deliberately excluded from `organization_service.py`'s `ROLE_HIERARCHY`/`_build_subtree` (which only spans Super Admin → Account Manager → Team Lead → Staff) — Site Lead has unconditional visibility of everything rather than a place in the reporting tree.

**This directory is `unified-frontend/`** (formerly `rbac-service/frontend/`, one level down inside a since-retired `rbac-service/` parent — see the root `CLAUDE.md`'s frontend-consolidation note). Next.js 16 (App Router) + TypeScript + Tailwind + Shadcn UI. Its FastAPI backend counterpart lives at the monorepo-root `unified-backend/app/rbac/` (see "Commands" below and the root `CLAUDE.md`'s "Backend consolidation" section) — the two used to be sibling `backend/`/`frontend/` subdirectories under one `rbac-service/`, now they're two independent top-level directories with no shared parent.

**Path note for the rest of this document**: everywhere below that says `frontend/...` (a holdover from when this file lived one level up, in `rbac-service/`) now means "relative to this directory" — e.g. `frontend/src/lib/api.ts` is just `src/lib/api.ts` from here. Not rewritten line-by-line throughout; translate mentally.

This service (this frontend, plus the `app/rbac/` half of `unified-backend`) is the **shell application**: it owns authentication, session, role-based routing/navigation, and layout for the whole product. It is also the **sole issuer** of the JWT access/refresh tokens `app/ticketing` verifies — now a module boundary inside one process rather than two separate ones, see root `CLAUDE.md`. Rotating `JWT_SECRET_KEY` in `unified-backend/.env` invalidates every session across both domains.

**Product branding is "UTMS" / "Unified Ticket Management System"**, not "RBAC" / "Enterprise Platform" — the sidebar (`components/layout/sidebar.tsx`), the mobile header (`components/layout/dashboard-layout.tsx`), and the `<title>` metadata (`app/layout.tsx`) were all renamed from the original RBAC-only branding once the product became the combined ticketing+RBAC system. "RBAC" still appears throughout this document and in code comments/identifiers (role names, internal module names, etc.) — that's the system's technical/historical name, not a stale branding reference; only the **user-facing** product name changed.

**The Ticket Management workspace is embedded here, not linked to.** `ticketing-service/frontend` was copied (not just referenced) into `src/ticket-workspace/` and mounted as a client-only `react-router-dom` subtree inside this Next.js app — see "Ticket workspace embedding" under Frontend architecture below. Staff, Team Lead, and Manager land there after login; Super Admin and Viewer instead see this app's own admin dashboard. The two frontends are **not** kept in sync automatically — a ticket-related UI change usually needs to be made in both places (`ticketing-service/frontend/...` and this directory's own `src/ticket-workspace/...`, same relative structure).

## Architecture

**Path note**: every `app/...` path below is relative to `unified-backend/app/rbac/`, not this repo's own (now-empty) `backend/`. `unified-backend/alembic_rbac/` is this domain's migration chain (not `backend/alembic/`).

### Backend: layered, but authorization stops at the frontend

Routes (`app/api/v1/`) → Services (`app/services/`) → Repositories (`app/repositories/`) → SQLAlchemy models. Two model sources are merged into one `Base.metadata`: `User` and `Role` come from `shared_models` (a local editable install both `app/rbac` and `app/ticketing` import — see root `CLAUDE.md`), while `Permission`, `RolePermission`, `AuditLog`, `UserPermissionOverride`, and `PermissionRequest` are local (`app/rbac/models/`). `unified-backend/alembic_rbac/env.py` (formerly `backend/alembic/env.py`) imports both sources and filters out `app/ticketing`'s tables via `include_object` so `alembic revision --autogenerate -c alembic_rbac/alembic.ini` only diffs this domain's own tables.

**Every route only checks authentication (`Depends(get_current_active_user)`), never authorization — with one deliberate exception (see below).** There is no generic `require_permission`-style dependency anywhere in the backend — any authenticated user can call almost any endpoint (list all users, change any role's permissions, etc.) regardless of their role or permission set. Everything else — permission-gated buttons, role-based page visibility, the Users page's reporting-hierarchy filtering — is still enforced **only in the frontend**. Treat this as a known architectural constraint, not something to silently "fix" by adding backend checks unless asked.

**The one exception is per-user permission overrides.** `POST/GET/DELETE /users/{id}/permission-overrides` (`api/v1/permission_overrides.py`) is the first real server-side authorization check in this backend — see "Per-user permission overrides" below for the full shape. Its check, `PermissionOverrideService.ensure_can_manage_overrides` (public — not prefixed with `_`, precisely because a second caller, `PermissionRequestService`, now reuses it directly rather than re-deriving the same role-scoping rules; see "Permission requests" below), was deliberately kept as a direct service-layer method rather than a generic dependency until there's a third real call site.

Audit logging is defined (`AuditLogService`, `audit_logs` table, `GET /audit-logs`) and **is now actually called** — by the permission-override grant/revoke flow (`PermissionOverrideService`) and the permission-request create/approve/reject flow (`PermissionRequestService`), which together are still its only writers. Every other user/role/permission mutation in this backend still leaves the table empty; don't assume a row exists for anything else without checking.

**Seeding is not purely additive** (`scripts/rbac_seed/seed.py`): the main loop only ever *adds* a permission grant a role is missing per `DEFAULT_ROLES` — it never revokes one just because a role's default list changed. Two explicit cleanup lists run once near the top of `seed()`, before the main loop, to handle everything the additive loop can't: `DEPRECATED_PERMISSIONS` (permission names to delete outright, including their `role_permissions` grants — e.g. `ticket:bulk_reassign`, `ticket:configure_routing`) and `REVOKED_GRANTS` (specific `(role_name, permission_name)` pairs to un-grant without deleting the permission itself — e.g. Staff's `ticket:create`). When narrowing a role's permissions going forward, add the removal to one of these two lists — editing `DEFAULT_ROLES` alone will not revoke anything from a DB that already has the old grant. There's also a one-time in-place role rename block (`Role.name` mutated directly, same `role_id`, guarded so it only fires if the old name exists and the new one doesn't yet) — this is the safe pattern for renaming a role without orphaning users/grants, used for the "Manager" → "Account Manager" rename.

**Database URL normalization** (`app/core/config.py`, `Settings.normalize_database_url`) is important to understand before touching `DATABASE_URL` handling: it rewrites `postgres://`/`postgresql://` → `postgresql+asyncpg://`, translates `sslmode=` → `ssl=`, and strips `channel_binding=` — all required because Neon's connection strings use kwargs asyncpg doesn't accept. `alembic/env.py` does the reverse translation (`+asyncpg` stripped, `ssl=` → `sslmode=`) to get a psycopg2-compatible sync URL for migrations. Don't naive-string-replace this logic — use `urllib.parse`, per the existing implementation.

### Per-user permission overrides

A role's permission bundle (`role_permissions`) is the default; `user_permission_overrides` (`app/models/permission_override.py`) is a small, purely **additive** exception list layered on top — granting one specific person a capability their role doesn't include, without touching the role or affecting anyone else who holds it. Soft-revocable (`revoked_at`/`revoked_by`, never hard-deleted) with an optional `expires_at` (blank = permanent), and a partial unique index (`WHERE revoked_at IS NULL`) so at most one *active* grant exists per user+permission at a time while a full grant→revoke→re-grant history is preserved.

**Overrides can optionally be scoped to one specific ticket** (`scope_ticket_id`, nullable UUID, no FK — see below for why). `NULL` (the common case, and the only kind this section originally described) means the grant applies everywhere the permission would normally apply; a non-`NULL` value means it applies *only* for that one ticket. This exists for exactly one workflow today: a Staff member requesting `ticket:editother_ticket` (see "Permission requests" below) scoped to one specific teammate's ticket, rather than a blanket grant to work every ticket in scope. The active-grant uniqueness index is `(user_id, permission_id, COALESCE(scope_ticket_id, <sentinel-uuid>)) WHERE revoked_at IS NULL` — the `COALESCE` matters: a plain 3-column unique index would treat every `NULL` as distinct from every other `NULL` (Postgres's default null-handling) and silently allow two simultaneous *global* grants for the same user+permission, which isn't what "at most one active global grant" should mean. `scope_ticket_id` is deliberately a bare UUID with no `ForeignKey` — `tickets` is owned by `app/ticketing`'s own migration chain (`alembic_ticketing`), and a cross-chain FK would tie the two histories together the same way the root CLAUDE.md's "never generate a migration in one chain that touches the other's tables" rule already warns against; validity is checked in application code instead (`PermissionRequestService.create_request` looks the ticket up via `app.ticketing`'s own `TicketRepository`, imported directly since both domains are one process now).

**`PermissionResolverService`** (`app/services/permission_resolver.py`) is the single place effective permissions are computed: `role_permissions(role_id) ∪ active_global_overrides(user_id)` for the flat list, plus a **separate** `scoped_permissions: dict[str, list[str]]` return value (permission name → ticket ids) built from active *ticket-scoped* overrides only. This split matters: a ticket-scoped override is deliberately excluded from the flat permission list, so a plain `has_permission(user, "ticket:editother_ticket")` membership check can never mistake "holds it for one ticket" for "holds it everywhere" — call sites that need to check a scoped grant use `app.ticketing.services.access_control.has_permission_for_ticket(user, permission_name, ticket_id)` instead, which checks both the flat list and the scoped map. `AuthService.get_current_user`/`login`/`refresh_token` all call `get_effective_permissions` and thread all three return values through — `permissions` and `scoped_permissions` both end up as separate JWT claims (see root CLAUDE.md's "Cross-service identity"), and `override_permissions` additionally appears on `/auth/me`'s `CurrentUser` response, alongside `scoped_permissions` itself (both nullable/optional on the frontend's `AuthUser` type, `src/types/index.ts`, and passed through as-is by `authService.me()`, `src/services/index.ts`). `ticketing-service`'s `TicketActions.tsx` reads `currentUser.scoped_permissions` to disable-in-place ticket actions the caller can't perform on that specific ticket — see that repo's CLAUDE.md. Never call `RolePermissionRepository.get_permissions_by_role` directly to answer "what can this user do" — that only tells you what their role grants, not their actual effective set.

**Grant/revoke authorization** (`PermissionOverrideService.ensure_can_manage_overrides`) is scoped by the actor's own role: Super Admin/Site Lead have unconditional authority; Account Manager is restricted to their own reports, resolved via `OrganizationService.get_subordinate_user_ids` (flattens the existing `_build_subtree` org-chart tree rather than duplicating the `manager_id`/`teamlead_id` traversal). `grant()` rejects a permission the target's role already includes (400 — "would be redundant") — this check is unaffected by scoping, since a ticket-scoped grant is for a permission the target's role does *not* hold at all (e.g. Staff and `ticket:editother_ticket`), never a permission they already have generally. Every grant/revoke writes a row via `AuditLogService.create_log` — see the audit-logging note above.

**Frontend**: `components/users/user-permission-overrides.tsx` renders a "Personal Permission Grants — {name} only" section inside `user-detail-drawer.tsx`, **below and visually distinct from** the existing role-permissions editor — the two are unrelated mutations (one edits the role everyone with it shares, the other edits one person) that were previously easy to conflate since only the role editor existed. Gated by `<PermissionGuard permission="permission:override_grant">`, same convention as everywhere else. The grant form filters out permissions already in the target's role bundle client-side (mirroring the backend's redundant-grant rejection) so the UI never offers a choice the API would reject.

### Permission requests

A self-service layer on top of the override mechanism above: `permission_requests` (`app/models/permission_request.py`) lets any user ask for a permission they don't currently have, addressed to a *role* (e.g. "Account Manager") rather than a specific person — any user holding that role can review it, subject to the exact same authority `ensure_can_manage_overrides` already enforces (Super Admin/Site Lead see and can act on every pending request regardless of which role it's addressed to; an Account Manager only sees/acts on ones from their own reports). This is deliberately a request/review *workflow* layered on top of, not a reimplementation of, the override mechanism — approving one calls `PermissionOverrideService.grant()` directly and stores the resulting `override_id` back on the request row (`granted_override_id`), so there is exactly one code path that ever creates a `UserPermissionOverride`, and a request's "is this now revoked" state is read through that link rather than duplicated.

**`PermissionRequestService.list_eligible_approver_roles`** answers "which roles could even grant this permission" by querying `role_permissions` **live** for every role (never `seed.py`'s static `DEFAULT_ROLES`, which can drift from the DB — see the seeding note above) and keeping only roles that hold both `permission:override_grant` and the requested permission itself — a role can't vouch for a permission it doesn't hold. This is also why Team Lead never appears as an approver option today: it holds neither, regardless of what CLAUDE.md or seed.py comments might suggest a future role matrix could look like.

**A request can optionally carry `scope_ticket_id`**, turning it into a ticket-scoped grant instead of a blanket one (see "Per-user permission overrides" above) — `approve()` passes it straight through to `PermissionOverrideService.grant()` unchanged, so there's still exactly one place a `UserPermissionOverride` gets created. Two purpose-built read-only endpoints back the request form's dynamic fields, both under `/permission-requests/scope/`:
- `GET .../staff-options` — the requester's teammates (other Staff sharing the same `teamlead_id`, via `UserRepository.get_by_teamlead` — the same lookup `OrganizationService._build_subtree` already uses for a Team Lead's org-chart children, reused rather than reimplemented).
- `GET .../ticket-options?staff_id=` — that teammate's assigned tickets, via `app.ticketing`'s own `TicketRepository.list_all(agent_id=staff_id)` filtered to `agent_id == staff_id` exactly (the repository method's own `agent_id=` branch deliberately also includes unassigned tickets for its original shared-pool-visibility caller, so the service layer re-filters). `staff_id` is validated as a genuine teammate first (`PermissionRequestService._is_teammate`) — this endpoint intentionally bypasses the normal category-scoped ticket-visibility rule (`ensure_agent_can_view_ticket`) that a regular ticket list would apply, since the actual security boundary here is "is this really your teammate," not category; don't reuse it as a general-purpose ticket-listing endpoint.

Both fields only ever appear in the frontend when the selected permission is `ticket:editother_ticket` — and since `eligiblePermissions` already excludes permissions the requester already effectively holds, and `editother_ticket` is a role default for everyone except Staff, in practice only Staff members ever see these fields at all (no separate role check needed to gate them).

**Frontend**: a full page at `/permission-requests` (`app/(dashboard)/permission-requests/page.tsx`) with two tabs — "My Requests" (visible to everyone) and "Pending My Review" (gated by `<PermissionGuard permission="permission:override_grant">`, hidden entirely otherwise). Reachable from **every** role's sidebar, including Staff/Team Lead/Account Manager — see the role-access.ts note below for why that's not the architectural fork it might look like. No push-notification system was built for this (the topbar bell is still the pre-existing UI-only mock, see below) — "notification" is satisfied by the pull-based "My Requests" tab reflecting the decision once you reload it, deliberately mirroring how `ticketing-service`'s Edit Access workflow already works.

### Frontend: App Router + centralized role-access config

- **Ticket workspace embedding**: `src/ticket-workspace/` is an adapted copy of `ticketing-service/frontend/src/`. `tsconfig.json` maps `@tw/*` → `./src/ticket-workspace/*` (separate from this app's own `@/*` → `./src/*`) so the two codebases' imports can never accidentally cross. `src/ticket-workspace/TicketWorkspaceApp.tsx` mounts the copied page tree under a `react-router-dom` `<BrowserRouter basename="/dashboard">`; `src/app/(dashboard)/dashboard/[[...slug]]/page.tsx` is the optional-catch-all Next.js route that renders it (via `next/dynamic(..., { ssr: false })` — `BrowserRouter` reads `window.location` at render time, which crashes Next's one server-render pass otherwise) for Staff/Team Lead/Account Manager (there is no role literally named `"Manager"` — see `ticketing-service/CLAUDE.md`'s "Roles" section if you see that string hardcoded anywhere, it's dead), `src/components/dashboard/site-lead-dashboard.tsx` for Site Lead (its own dedicated dashboard, not routed into the ticket workspace — Site Lead's day-to-day is org oversight, not hands-on ticket work), or `src/components/dashboard/viewer-dashboard.tsx` (this app's own original admin dashboard, pre-embedding) for Viewer and Super Admin. `src/ticket-workspace/context/AuthContext.tsx` is a thin adapter re-exposing this app's own `useAuthStore` instead of holding its own session state — the embedded copy has no login/token logic of its own; auth is exclusively this shell app's responsibility. `src/ticket-workspace/components/layout/AppLayout.tsx` similarly only renders a per-page title bar via this app's own `PageHeader` — Sidebar/Topbar/AuthGuard all come from this app's own `DashboardLayout`, one level up.
- **Design-token unification**: `.tm-scope` (`src/app/globals.css`) remaps the ticket workspace's own CSS variables (canvas/surface/accent/warning/danger/success/info/teal, plus the `slate`/`border`/`muted` scale) onto this app's actual computed dark-theme values (converted once from its HSL shadcn tokens) so the workspace and this app's own pages (Profile/Settings/Users/Roles) render as one consistent neutral-dark design system, not two visually distinct products. This applies to every role that reaches the workspace (Staff/Team Lead/Manager) uniformly — there is no per-role theme branching. Extending Tailwind's `content` glob to include `src/ticket-workspace/**/*` in `tailwind.config.ts`, plus a handful of non-colliding token additions there, is what makes the copied components' utility classes compile at all.
- `src/lib/role-access.ts` is the single source of truth for role-based UI behavior: `ROLE_NAMES` constants, per-role sidebar visibility (`NAV_ITEMS_BY_ROLE`, including which roles see the ticket-workspace nav items vs. this app's own Users/Roles vs. both — e.g. "Create Dummy Mail" is included for `ACCOUNT_MANAGER` but not `TEAM_LEAD`/`STAFF`), `SUPERVISOR_ROLE_NAMES`/`isSupervisorRole()`, and which roles a given role is allowed to assign on user creation (`CREATABLE_ROLES_BY_ROLE`). **`SUPERVISOR_ROLE_NAMES` here is only `[SITE_LEAD, SUPER_ADMIN]` and does *not* actually mirror `ticketing-service`'s real backend role set** (`{"Team Lead", "Account Manager", "Site Lead", "Super Admin"}`, in that repo's `access_control.py`) — it happens to match `ticketing-service/frontend`'s own same-named-but-equally-narrow `SUPERVISOR_ROLES` constant, but neither frontend constant matches the backend's actual set. This is known, pre-existing drift, not yet reconciled — don't assume any two of these three role-sets agree without re-reading all three. New role-conditional UI logic should extend this file rather than hardcoding role-name strings in components.
- **Two different nav items both currently *display* the label "Audit Logs" — they are still two entirely separate systems, only the label collided.** The `NavItemKey` internally named `"Ticket Audit Log"` (the ticket-workspace's own per-ticket compliance trail, `ticket_audit_logs` table, `app.ticketing`) was relabeled from "Ticket Audit Log" to "Audit Logs" via its `nav.ticketAuditLog` translation value only — the `NavItemKey` string itself, its route (`/dashboard/audit-logs`), and its translation *key* name were deliberately left unchanged, specifically to avoid colliding with the other, differently-keyed `NavItemKey` literally named `"Audit Logs"` (the RBAC-native `audit_logs` table, `app.rbac`, route `/audit-logs`) — renaming the key itself would have made two `menuItems` entries share one title string and both render side-by-side under an ambiguous label. **The RBAC-native "Audit Logs" nav item was also removed from every role's `NAV_ITEMS_BY_ROLE` array** (previously Super Admin/Site Lead only) — the page, its route, and its API are untouched and still reachable directly (e.g. the Super Admin dashboard's own "Latest Audit Logs" card still links straight to `/audit-logs`), this only removed the sidebar entry point. Net effect: the sidebar shows exactly one "Audit Logs" item per role today (the ticket-workspace one, for agent roles), but don't assume that's structurally guaranteed — if the RBAC-native one is ever re-added to `NAV_ITEMS_BY_ROLE` for some role, that role's sidebar would show two rows both labeled "Audit Logs" pointing at different pages.
- **RBAC's own `audit_logs` table now has substantially wider write coverage than "permission-override grant/revoke and permission-request create/approve/reject" (see the Architecture section's older note below, which describes the original, narrower state)**: `AuthService` now also writes `auth.login` / `auth.login_failed` (with a `reason` — `invalid_email`/`account_inactive`/`invalid_password`) / `auth.logout` (a new `POST /auth/logout` route — tokens are stateless JWTs with no server-side session, so this is audit-only) / `auth.change_password`, all capturing the caller's IP where available; `UserService` writes `user.create`/`user.update`(a diff of just the changed fields)/`user.delete`/`user.activate`/`user.deactivate`/`user.role_changed` (a separate row from `user.update` specifically when `role_id` changes, so role-history can be filtered without parsing every update's old/new-value JSON); `RoleService` writes `role.create`/`role.update`/`role.delete`; `RolePermissionService.replace_permissions` diffs the previous permission set against the new one and writes `role.permissions_added`/`role.permissions_removed` (only for what actually changed, never a blanket re-log of the whole set). The `/audit-logs` page's own Status column is now real too (derived from the action name containing `failed`/`reject`, not hardcoded to always show "Success"), and it resolves + displays each row's Role (joined client-side against `roleService.list()`, same pattern as the existing user-name join) in addition to User/Action/Entity/Timestamp/IP. Deliberately still NOT logged here (checked and ruled out as out of scope, not overlooked): attachment download/delete and mail draft save/delete would need new `ticket_audit_logs` `AuditEventType` values — i.e. a new Postgres-enum-widening migration — Reports/Settings have no backend endpoint at all to hook into, and Ticket Resolved/Closed/Reopened are already fully covered by the existing `STATUS_CHANGED` event.
- **`app/core/rbac_cache.py` (`unified-backend`) is an in-memory, per-process cache keyed on `(user_id, permission_version)`** that lets `get_current_user` skip a full effective-permissions recompute on every request while a session's cached entry is still valid (`rbac_cache_ttl_seconds`, default 30s; `rbac_cache_max_size`, an LRU-evicted per-process cap). `User.permission_version` (a plain integer column, default `1`) is the invalidation key: it's bumped on the affected user's own row whenever their `role_id`/`category_id`/`manager_id`/`teamlead_id`/`is_active` changes or a personal permission override is granted/revoked (see `UserService.update_user`/`activate_user`/`deactivate_user`, and `PermissionOverrideService`), and bumped in bulk (one `UPDATE ... WHERE role_id = :id`, not a per-row Python loop) for every user holding a role whose own permission set changes (`UserRepository.bump_permission_version_for_role`, called from `RolePermissionService.assign_permission`/`remove_permission`/`replace_permissions`). A JWT's own claimed `permission_version` no longer matching the DB's live value is what silently rejects a stale cached session on its next DB-verified request — nothing scans or actively invalidates the cache itself. `test_rbac_cache.py`/`test_get_current_user_cache.py` (`unified-backend/tests/`) cover this.
- **`NAV_ITEMS_BY_ROLE` is one shared sidebar config for every role, not two separate sidebars.** It's easy to misread "Staff/Team Lead/Account Manager land on the ticket workspace after login" (above) as meaning those roles never see this app's own RBAC-native pages, but `src/components/layout/sidebar.tsx`'s `SidebarContent` renders from the exact same `menuItems`/`NAV_ITEMS_BY_ROLE` regardless of role — it just mixes ticket-workspace routes (`/dashboard/inbox`, `/dashboard/tickets`, ...) and RBAC-native routes (`/users`, `/roles`, `/permission-requests`, ...) in one list, filtered per role by `canSeeNavItem`. So a *new* RBAC-native page only needs one addition here (`NavItemKey` + `NAV_ITEM_TRANSLATION_KEY` + a `menuItems` entry in `sidebar.tsx`) to become reachable from every role's sidebar, including the ones that land in the embedded ticket workspace by default — no separate ticket-workspace-embedded page or cross-service API call is needed just to reach it. `Permission Requests` (see above) is the first page built this way.
- **The topbar notification bell (`top-navbar.tsx`) is real, not a mock** — backed by `app/notifications` (`GET /notifications`, `POST /notifications/{id}/read`, `POST /notifications/read-all`), polled every 30s. `notify()` (`unified-backend/app/notifications/service.py`) is the single write path every trigger across both `app.rbac` and `app.ticketing` calls through — permission requests/overrides, ticket assignment, mail received/replied, edit-access requests, and (see root CLAUDE.md's "SLA & Escalation") SLA at-risk/breach/escalation all fan out through it. `SLA_AT_RISK`/`SLA_BREACHED`/`SLA_ESCALATED` additionally get a toast (not just a bell update) via a `SLA_NOTIFICATION_DOT` membership check in `top-navbar.tsx`. **Every notification's `link` is written by the backend as if the ticket workspace were mounted at the app root** (`/tickets/{id}`, `/inbox`) — because that's where it lives in the standalone `ticketing-service/frontend` app; in this unified app it's mounted under react-router's `basename="/dashboard"` instead, so pushing the raw link 404s. `resolveNotificationHref()` in `top-navbar.tsx` prefixes only the paths that actually belong to that embedded subtree (`/tickets`, `/inbox`, `/interactions`, `/create-mail`, `/audit-logs`) with `/dashboard`, leaving RBAC-native root links (`/permission-requests`) untouched — this was a real, previously-live 404 bug; if a new notification type's `link` still 404s, check whether its target path needs adding to that prefix list. Clicking a notification's own X removes only that one (client-side only — there's no delete/dismiss endpoint, dismissed ids are tracked in a ref and filtered back out of every subsequent poll) and never navigates.
- **Permissions include a Ticket Management group**: `unified-backend/scripts/rbac_seed/seed.py`'s `DEFAULT_PERMISSIONS` includes the `ticket:*`/`communication:*` permission catalog (create, view_own, view_others, assign, transfer, change_priority, change_category, escalate, `editown_ticket`, `editother_ticket`, etc.) assigned per-role to mirror the ticket workspace's actual capability matrix — `ticket:bulk_reassign`, `ticket:configure_routing`, and (as of the `editown_ticket`/`editother_ticket` split) the old single `ticket:edit_ticket` were all removed outright (see `DEPRECATED_PERMISSIONS` in `seed.py`, which deletes the rows and their grants rather than leaving them orphaned); don't reintroduce any of the three without checking whether that removal reasoning still applies. `src/components/users/user-detail-drawer.tsx` renders them as a "Ticket Management" card via the same prefix-based grouping (`permission_name.split(":")[0]`) used for `user:`/`role:`/`audit:` permissions. **These used to be entirely self-contained bookkeeping** (`app.ticketing` authorized purely by role name, not these rows) **but are now partially real**: the JWT's `permissions`/`scoped_permissions` claims (see the root CLAUDE.md's "Cross-service identity" section) propagate a user's effective permission set to `app.ticketing`, which enforces a growing subset of them server-side via `ensure_has_permission`/`has_permission_for_ticket` — currently `ticket:change_priority`, `ticket:transfer` (Staff-only, layered on top of the pre-existing role-based supervisor check), and the `ticket:editown_ticket`/`ticket:editother_ticket` pair (see `ticketing-service/CLAUDE.md`'s "Permission-based enforcement" and "Edit access requests" sections). Most of the catalog is still unenforced bookkeeping — grep `unified-backend/app/ticketing/services/access_control.py` for the current, real list of `ensure_has_permission`/`has_permission_for_ticket` call sites before assuming any given `ticket:*` permission actually gates anything.
- `src/components/auth/AuthGuard.tsx` gates the whole `(dashboard)` route group on `authService.me()` resolving; `PermissionGuard` conditionally renders based on `useAuthStore().hasPermission()`. Individual pages (e.g. Users) additionally self-gate with a role allowlist check that renders `AccessDenied` (403) for disallowed roles — this is a page-level pattern, not a route middleware, since `middleware.ts` only handles public/auth routing, not per-role authorization.
- Users page hierarchy filtering is entirely client-side: fetch all users (`page_size: 100`), then filter in-memory by `manager_id`/`teamlead_id` against the current user, since the backend has no server-side filtering for this and no row-level authorization (see above).
- Permission editing lives in `src/components/users/user-detail-drawer.tsx` (a right-side `Sheet`), not a standalone page — there is no `/permissions` route. Opening a user's drawer shows **two separate** permission-editing surfaces: the role-permissions editor (`PUT /roles/{role_id}/permissions`, edits **every** user who holds that role) and, below it, the per-user "Personal Permission Grants" section (`user-permission-overrides.tsx`, edits only this one person — see "Per-user permission overrides" above). A Manager editing role permissions can only toggle permissions they personally hold (checked against their own `AuthUser.permissions`); Super Admin is unrestricted; Team Lead/Staff/Viewer can't reach either surface at all.
- i18n is a small custom system, not a library: `src/lib/i18n/translations.ts` (flat dictionary keyed by dotted strings, one object per language) + `src/hooks/use-translation.ts` (reads the active language from `useSettingsStore`, which persists via zustand). Adding a UI string requires adding the same key to every language dictionary — `Record<keyof typeof en, string>` on the non-English dictionaries enforces this at compile time.
- `src/components/shared/data-table.tsx` (`DataTable`/`DataTablePagination`) wraps `@tanstack/react-table` and is reused by both Users and Audit Logs; it supports an optional `onRowClick` for row-level navigation/drawers without needing per-page reimplementation.
- Zustand stores (`src/store/`) are the persistence layer for anything not backed by the API: `auth-store.ts` (current user + theme), `settings-store.ts` (language, notifications, security prefs, mock session list), `profile-extras-store.ts` (phone/address/avatar URL — fields with no backend column, stored client-side only and not synced across devices).
- Backend API base URL is `process.env.NEXT_PUBLIC_API_URL` (`src/lib/api.ts`), falling back to `http://localhost:8000/api/v1` for local dev. Being a `NEXT_PUBLIC_*` var, it's baked in at build time — changing it in a deployed environment requires a rebuild, not just a restart. The embedded ticket workspace's **separate** axios instance (`src/ticket-workspace/api/client.ts`) uses its own `NEXT_PUBLIC_TICKETING_API_URL`, falling back to `http://localhost:8001` — a stale default from before the backend merge, see the matching Known Issues entry below.

## Known Issues

- `npm run lint` is broken: Next.js 16 dropped `next lint`, but the project still has an old-format `.eslintrc.json` rather than an ESLint 9 flat config. Use `npx tsc --noEmit` to check for real errors instead.
- `docs/ARCHITECTURE.md`, `docs/API.md`, and `docs/DEPLOYMENT.md` describe an earlier/aspirational design that has drifted from the actual implementation (e.g. they document a `refresh_tokens` table, `require_permission()` backend enforcement, and extra `/users` query params like `role_id`/`is_active` — none of which exist in the current code). Don't treat `docs/` as authoritative; verify against the actual source.
- **Turbopack workspace-root inference**: without the `turbopack.root` pin in `next.config.mjs`, Next.js 16's Turbopack walks up the filesystem looking for a lockfile to infer the project root and can land on an unrelated one several directories up (e.g. a stray `package-lock.json` in a parent/home directory) — this 404s every single route with no obvious error. `next.config.mjs` already pins `turbopack.root` to `__dirname` to prevent this; if every route suddenly 404s again after a config change, verify that pin is still intact before debugging anything else. Separately, a `next dev` process that's been running across many long sessions (especially through `npm install`/`uninstall` churn in the same directory, e.g. temporarily installing Playwright for verification) can land in a similar all-routes-404 state even with the config correct — a clean restart (kill the process, `npm run dev` again) resolves it; this is a stale-process issue, not a config regression, and is worth ruling out first.
- **`get_settings()` is `@lru_cache`d** (`backend/app/core/config.py`) — `.env` changes while the backend is already running have no effect until the process is restarted (`uvicorn --reload` only restarts on Python file changes, not `.env` edits). A config-dependent error that persists after editing `.env` almost always means "restart the server," not "the code is wrong." Same gotcha applies to `ticketing-service`'s backend.
- **A stale backend process can look exactly like "you have no eligible data" instead of an error.** A brand-new route added to a running `uvicorn` process that never picked up the change (see the `--reload` gotcha above) 404s; if the calling frontend code doesn't distinguish `isError` from "the query succeeded and genuinely returned an empty list," a 404 renders as a plausible-looking empty state instead of a visible failure — this happened for real with the Permission Requests dialog (rendered "You already have every permission" when the backend was simply missing the route). Any new React Query-backed dropdown/list should branch on `query.isError` explicitly, not just `query.isLoading` vs. "array is empty" — and if a brand-new page's data looks suspiciously, uniformly empty right after adding backend routes, check `GET /openapi.json` against the port the frontend is actually calling before assuming the frontend or the business logic is wrong.
- **The embedded ticket workspace's stale `:8001` default silently network-errors every ticketing request while RBAC's own requests keep working.** After the backend merge (see root CLAUDE.md), nothing listens on port 8001 anymore — everything is one process on 8000. `src/ticket-workspace/api/client.ts`'s `NEXT_PUBLIC_TICKETING_API_URL || "http://localhost:8001"` fallback was never updated, so an unset env var means every `tickets`/`inbox`/`interactions` call fails with a browser network error (visible in DevTools as `(failed) net::ERR_CONNECTION_REFUSED` or similar) while `notifications`/`users`/`auth` calls succeed — a confusing, asymmetric failure that looks like a backend bug but is a frontend config issue. Fix: add `NEXT_PUBLIC_TICKETING_API_URL=http://localhost:8000` (root, no `/api/v1` — see "Backend consolidation" in root CLAUDE.md for why ticketing routes are unprefixed) to `.env.local` (this directory's own), then restart `npm run dev` (`NEXT_PUBLIC_*` vars are baked in at server start, not hot-reloaded).
- **A dependency declared in `package.json` isn't necessarily installed** — the Mail rich-text composer (`src/ticket-workspace/components/mail/RichTextEditor.tsx`, `@tiptap/*` packages) was added to `package.json`'s `dependencies` without `npm install` ever being re-run afterward here, so `node_modules/@tiptap` was simply missing. Next.js's resulting error (`Module not found: Can't resolve '@tiptap/react'`, repeated for every `@tiptap/*` import, plus a same-shaped 500 on every route that imports it transitively) reads like a real code/config bug, not a "you forgot to install" one. If a module-not-found error names a package that's clearly already used elsewhere in the file/repo, check `node_modules/<package>` before assuming the import path or the code is wrong — `npm install` (no arguments needed, it reads `package.json`) resolves it, then restart `npm run dev` (a newly-installed package isn't picked up by hot-reload).
- **Turbopack's persistent `.next/` cache is not relocatable across a directory move/rename.** When this directory was renamed from `rbac-service/frontend` to `unified-frontend` (see the top-of-document note), the `.next/` build cache moved along with it — but Turbopack's cache stores absolute paths tied to the *old* location, so `npm run dev` afterward panics repeatedly with `FATAL: An unexpected Turbopack error occurred` / `Next.js package not found` (a `next-panic-*.log` gets written to `%TEMP%` each time). The `turbopack.root` pin in `next.config.mjs` does **not** cover this — that only fixes workspace-root *inference*, not a stale cache's baked-in paths. Fix: stop the dev server, `rm -rf .next`, restart `npm run dev` — do this any time this directory (or its parent) is moved/renamed again, before debugging anything else.
- **A committed `.env` file and two committed `*.tsbuildinfo` build-cache files were found during a repo-wide audit — in `ticketing-service/frontend`, not this directory** (this directory's own `.gitignore` already correctly excludes both patterns). See that repo's `CLAUDE.md` for the specific files; not yet cleaned up as of this note.
- **`src/ticket-workspace/api/notifications.ts` is dead code in this copy** — zero importers anywhere in `src/ticket-workspace/` (confirmed by grep). Its counterpart in the standalone `ticketing-service/frontend` app *is* used (by that app's own `Topbar.tsx`), but the embedded copy relies on the shell's own top-navbar bell instead (itself a UI-only mock, see above) — this file is leftover from the initial whole-tree copy, never wired up here, and not currently imported by anything. Safe to delete unless a future task actually needs it.
- **`NEXT_PUBLIC_TICKETING_FRONTEND_URL` is a dead env var** — documented in `.env.example` and `render.yaml` (as backing a "sidebar 'Ticket Workspace' opens in a new tab" link), but no component anywhere in this codebase actually reads it; `components/layout/sidebar.tsx` routes to the ticket workspace via plain internal Next.js links instead, confirmed by grep. Don't assume setting it does anything.
- **Running `npm run build` (a production build) in this directory while `npm run dev` is already active for it corrupts the shared `.next/` cache and 404s every route.** Both processes write into the same `.next/` output directory, and Turbopack's dev manifests and a production build's own artifacts don't coexist there — the symptom looks identical to the two Turbopack `.next` entries above (every route 404s, e.g. hitting `/dashboard` right after login, with chunk requests intermixed with the failing document request), but the trigger here is different: it's not a directory rename or workspace-root misdetection, it's two Next.js processes sharing one output directory at once. Same fix as those two entries: stop the dev server, delete `.next/`, restart `npm run dev` — confirmed this resolves it (both `/login` and `/dashboard` returned 200 immediately after). Avoid running `npm run build` here while this directory's own `npm run dev` is running; if you need to validate both, stop the dev server first or build against a separate checkout.
- **An unhandled backend 500 reads in the browser as a plain CORS failure, not a 500 — this has happened twice, both fixed, but the pattern will recur.** An unhandled exception's response has no `Access-Control-Allow-Origin` header at all (compare against a normal 401, which correctly carries CORS headers), so the browser reports "blocked by CORS policy" / `net::ERR_FAILED` instead of surfacing the real error. If a request that used to work suddenly reads as a CORS error with no CORS config having changed, suspect an unhandled backend exception first — reproduce the exact service-layer call directly in a throwaway script (bypassing FastAPI/HTTP entirely) to get the real traceback, rather than debugging CORS headers. Two confirmed instances: (1) `GET /tickets/audit-logs` 500'd because the Postgres-native `audit_event_type_enum` type already had 8 SLA/escalation values (`SLA_MANUALLY_PAUSED`/`SLA_MANUALLY_RESUMED`/`SLA_BREACH_DETECTED`/`SLA_ESCALATED`/`ESCALATION_CREATED`/`ESCALATION_ACKNOWLEDGED`/`ESCALATION_ADVANCED`/`ESCALATION_CLOSED`) that the Python `AuditEventType` enum didn't yet — SQLAlchemy raised a `LookupError` deserializing any row using one, fixed by adding the missing members to the Python enum (see root CLAUDE.md's "SLA & Escalation" section for why these values later became genuinely live rather than legacy). (2) a missing `selectinload` import in `unified-backend/app/rbac/repositories/user_repository.py` 500'd `GET /users` (and five other methods — `get_by_role`, `list_active_by_role_name`, `get_by_manager_and_role`, `get_by_teamlead`, `get_by_category`) with a `NameError`, fixed by adding `selectinload` to the existing `from sqlalchemy.orm import joinedload` line; scanned the rest of the backend for the same missing-ORM-loader-import shape afterward and found no other instances.
- **Every popup/dialog in the app closes only via its own explicit Cancel/Close/X control — never by clicking outside it or pressing Escape.** This is enforced once, centrally, not per call site: `src/components/ui/dialog.tsx`'s `DialogContent` and `alert-dialog.tsx`'s `AlertDialogContent` both override `onPointerDownOutside`/`onInteractOutside`/`onEscapeKeyDown` (radix's `AlertDialogContent` type omits the first two — it already hardcodes them internally, only `onEscapeKeyDown` needs the override) to `event.preventDefault()`, placed *after* `{...props}` so no individual call site can reintroduce either close path by passing a conflicting handler; `sheet.tsx`'s `SheetContent` gets the same three. The ticket-workspace's own hand-rolled `Modal.tsx` simply has no `onClick` on its backdrop and no Escape-key listener at all (rather than a prop to override). Two of its own hand-rolled slide-over drawers, `InteractionDetailsDrawer.tsx`/`AuditLogDetailsDrawer.tsx`, needed the same treatment (a backdrop `onClick={onClose}` and an Escape `keydown` listener, both removed). If you add a new modal/drawer/popover anywhere in this app, it should not need its own lockdown code — reuse one of these existing wrapper components rather than hand-rolling another dismiss mechanism that would need the same fix applied a second time.

## Service Unification (RBAC + Ticket Management)

## Per-role dashboards and Reports (Account Manager / Team Lead / Staff)

Account Manager, Team Lead, and Staff now land on **the same dashboard layout as Super Admin** at the bare `/dashboard` root (previously they had no RBAC-native dashboard at all — they fell straight through to `TicketWorkspaceApp` regardless of slug; any deeper slug like `/dashboard/inbox` still does). `app/(dashboard)/dashboard/[[...slug]]/page.tsx` gained three more `role === ... && slug.length === 0` branches alongside the existing Super Admin/Site Lead/Viewer ones.

**Reuse, not duplication**: `components/dashboard/super-admin-dashboard.tsx`'s `SuperAdminDashboard` took two new optional props, `tickets?: MockTicket[]` (defaults to `MOCK_TICKETS`) and `description?: string` — every KPI/chart/list inside it was already a pure derivation of a `tickets` array, so making it reusable needed no other change, and Super Admin's own rendering is byte-identical to before. Three thin wrapper components — `account-manager-dashboard.tsx`, `team-lead-dashboard.tsx`, `staff-dashboard.tsx` — each read `useAuthStore`, compute a role-scoped ticket subset, and render `<SuperAdminDashboard tickets={...} description="..."/>`. The Reports page (`app/(dashboard)/reports/page.tsx`) applies the identical scoping to the same `useMockTicketsStore` data before feeding it into `getReportMetrics`/`getCountsByXxx`, so CSV/Excel/PDF exports are scoped too; "Reports" was added to `NAV_ITEMS_BY_ROLE` (`lib/role-access.ts`) for all three roles (previously Super Admin/Site Lead only) — the page itself still has no role-based access gate of its own (unlike `/users`), so reachability is nav-visibility only, not enforced.

**The scoping helpers are a deliberate workaround, not real ownership data — this is worth understanding before extending it.** `lib/mock-tickets.ts` gained `getTicketsForAccountManager(userId, tickets?)`, `getTicketsForTeamLead(userId, tickets?)`, and `getTicketsForStaff(userId, tickets?)`. `MockTicket` has no id-based owner field at all (only free-text `assignedTo`/`client` names drawn from this file's own fictional roster — `AGENTS`/`CLIENTS`), and none of those fictional names ever match a real seeded/created RBAC user's actual name. Matching by literal name (the obvious first approach) would show an **empty** dashboard for every real account. Instead, each helper deterministically buckets a real user — by hashing their own `user_id` (`stableIndex`, a simple non-random hash) — onto one fixed slice of the fictional data: Staff gets one fictional agent's tickets, Team Lead gets one of the three `AGENT_TEAM` teams, Account Manager gets a fixed-size slice of the client roster. This is stable and non-empty per user across reloads, but **it is not a real permissions filter** — two different real Team Leads could get the same team's data if their user_ids hash the same, and it has no connection to the real `manager_id`/`teamlead_id`/`category_id` hierarchy on the actual `User` model. Swap this for a real query once the ticketing backend has an aggregation endpoint to scope by (same "not real yet" caveat `SuperAdminDashboard`'s own top-of-file comment already carries for the mock dataset in general).

## Performance pass: round-trip collapsing across every major ticket-workspace page

A multi-session performance investigation started from a single complaint ("Interactions tab takes ~30s to load") and ended up establishing one reusable pattern — `list_visible_page`-style methods that JOIN visibility + display-name enrichment + `COUNT(*) OVER()` into one SQL statement — now applied to every major list endpoint in `app.ticketing`. The root cause, confirmed via a temporary `Server-Timing`-header instrumentation pass (`app/core/request_timing.py`, since left in place as permanent lightweight diagnostics) rather than guesswork: this environment's DB round-trip latency to Neon is ~300–700ms **per statement**, so a request's real cost is dominated by *how many separate SQL statements it issues*, not by any single query's own execution time (confirmed via `EXPLAIN (ANALYZE, BUFFERS)` showing sub-millisecond execution on every query touched below, at current dev-data scale). The fix pattern is always the same: stop pre-resolving a visible-ticket-id list (or an owned-client-id list) into Python and re-filtering by `IN (...)`; instead JOIN straight to `tickets`/`clients` and push an `account_manager_id`-owned-clients check down as an `IN (SELECT client_id FROM clients WHERE account_manager_id = ...)` subquery in the same statement.

**`GET /tickets/interactions`** (`InteractionRepository.list_visible_page`) went from 9 round trips → 2 (auth + one combined query): `UserRepository.get_by_id` switched from three `selectinload`s (user/role/category, 3 round trips) to one `joinedload` (many-to-one on both sides, no row-fanout risk); the interactions list itself now JOINs `tickets`/`clients`/`users` directly instead of pre-fetching a visible-ticket-id list and running a separate name-enrichment `UNION ALL` and a separate `COUNT(*)`. `pool_pre_ping` was deliberately **kept enabled** despite costing ~600–700ms/request on its own (measured via an A/B test) — this is a Neon pooler endpoint, exactly the scenario `pool_pre_ping` exists to protect against, and trading reliability for latency wasn't this session's call to make unilaterally. The same investigation fixed a real, separate functional bug: `TicketTimeline.tsx` (the per-ticket Timeline tab) had **no click handler at all** — a threaded reply's parent/children were only ever visible from the standalone Interactions tab, never from a ticket's own page — now wired to the same `InteractionDetailsDrawer` + `GET /interactions/{id}/thread` pattern.

**The same `list_visible_page` shape was then applied to every other major list/dashboard endpoint**, each with its own round-trip count collapsed and its own before/after cross-checked against independent raw SQL (not just trusted):

- **`GET /tickets` (Ticket List)** — previously fetched **every visible ticket unbounded** and did all filtering/sorting/tab-switching/pagination in the browser (`TicketsListPage.tsx`); server pagination existed in the backend's own docstrings but the frontend never actually sent `limit`. Now real: `view=pool|mine|all`, `ticket_type`, `sort_by`/`sort_dir` (validated against a fixed column allowlist, never interpolated into `order_by`) all apply in SQL before `LIMIT`/`OFFSET`, and `TicketRepository.list_visible_page` joins `client_name`/`client_company_name`/`agent_name`/`created_by_name` directly (three separate aliases of `users`, since a ticket references three distinct user rows at once) instead of `TicketService._attach_names`'s separate round trip. A new `GET /tickets/view-counts` (one grouped `FILTER`-clause query, same idiom as `InteractionRepository.count_by_view` on the Mail page) backs the three tab badges without fetching any tab's full row set.
- **`GET /tickets/audit-logs`** — `TicketService.list_all_audit_logs` used to call the unbounded `ticket_repository.list_all` (no `limit`) on **every single call, including this page's own every-15-second poll** — the same unbounded-visible-ticket-list mistake the Interactions endpoint had, just not yet fixed here. Now JOINs `ticket_audit_logs` straight to `tickets` for both visibility and `ticket_title`, and `search` (ticket title) is a real `ILIKE` in SQL instead of a Python substring filter over a now-removed in-process title dict.
- **`GET /tickets/dashboard-stats`** (new) — the ticket-workspace `Dashboard.tsx` used to call `listTickets()` (every visible ticket, unbounded) purely to derive eight stat-card counts (`assignedCount`/`openCount`/`slaRiskCount`/etc.) client-side. Replaced with one grouped query (`TicketRepository.dashboard_stats`, same `FILTER`-clause shape) plus two small bounded lists (`recent_tickets`: 6 most-recently-updated regardless of status; `critical_tickets`: HIGH-priority + open, bounded to 20 candidates and trimmed to 5 — a second `status IN (...)` condition wasn't worth adding to `list_visible_page` for this one caller).
- **`GET /tickets/{id}`** — `_attach_related_tickets` had a real N+1 (`ticket_repository.get_by_id` looped once per related ticket id, its own docstring admitted as much) — now one `list_by_ids` batch call. `TicketActions.tsx` and `EditAccessPanel.tsx` independently fetched the identical `GET /tickets/{id}/edit-access` on every ticket mount; the list now lives once in `WorkflowContext` (alongside `activeTicket`/`timeline`) and both components read it. `TicketDetailPage.tsx`'s `refreshAll` no longer awaits the timeline/edit-access fetches together with the ticket fetch — the ticket's own fields render the moment its own request resolves.
- **Mail (`useMailInbox.ts`)** — already the best-optimized page found (bounded `GET /inbox` pagination, lazy thread bodies, mutation-scoped refreshes) — the one real gap was `getInbox`/`getInbox`-calling `fetchBaseTab`/`loadMoreBaseTab` never passing the `AbortSignal` their own API wrapper already accepted; now a per-base-tab `AbortController` map cancels a superseded request instead of letting a stale one race a newer one to `setRowsByTab`.
- **RBAC-native Users/Roles/Audit-logs pages** (`unified-frontend/src/app/(dashboard)/...`) — all three independently issued the byte-identical `GET /users?page=1&page_size=100` under three different TanStack Query cache keys (`users-table`/`users-for-roles`/`users-for-audit`); unified to one shared key so the 30s `staleTime` window (`providers/query-provider.tsx`) actually dedupes them, which as a side effect also fixed a latent staleness bug — mutating a user from the Users page previously never invalidated the other two pages' stale copies.

**Deliberately not fixed, and why**: the RBAC Users/Roles/Audit-logs pages' `page_size=100` ceiling means hierarchy filtering, search, and date-range filters on those three pages all silently operate on only the newest 100 rows — a real correctness gap at scale, not just a speed one, but fixing it needs server-side hierarchy filtering plus a real pagination UI, a bigger change than this pass's scope (see the Users-page bullet under "Frontend: App Router + centralized role-access config" above, which already flags the client-side-only hierarchy filtering as a known constraint). The standalone `ticketing-service/frontend` was not touched by any of the above — consistent with the pre-existing, documented pattern (root CLAUDE.md's "The ticket workspace exists in two places") that only the embedded copy gets iterated on. RBAC's own dashboards (`super-admin-dashboard.tsx` and its per-role wrappers) were checked and confirmed to still be pure `MOCK_TICKETS` derivations with no real backend query to optimize (see "Per-role dashboards and Reports" above). One infrastructure fact surfaced but not acted on: Neon is `us-east-1`; the root `render.yaml` sets no `region:` for `unified-backend`, which defaults to Render's Oregon region — a real, unaddressed cross-region hop in production, reported but not changed.

A new Alembic migration (`alembic_ticketing/versions/1a3c5e7f9b2d_add_ticket_list_indexes.py`) added three indexes for query shapes that only became real, executed queries once the fixes above landed (a plain B-tree index can never serve a leading-wildcard `ILIKE`, hence the trigram one): `ix_tickets_updated_at` (the ticket-list page's default sort), a partial `ix_tickets_pool_view` (`WHERE agent_id IS NULL AND current_status = 'OPEN'`, covering the Open Pool tab's exact filter+sort), and a pg_trgm GIN index on `tickets.title` (the now-real search box; `pg_trgm` itself was already enabled by an earlier interactions-search migration). Verified upgrade/downgrade both ways; at current dev-table sizes Postgres still correctly prefers sequential scans over all three (same documented caveat as the pre-existing `interactions.subject` trigram index) — they exist for when the tables grow large enough for the planner to prefer them, not for today's row counts. Every changed endpoint's returned totals were cross-checked against independent raw SQL for multiple roles (Super Admin/Site Lead unrestricted, Account Manager client-owned, Team Lead/Staff category-scoped) rather than just type-checked; the full backend `pytest` suite (25 tests, including the 6 thread-reconstruction tests from the Interactions-tab fix) and `npx tsc --noEmit`/`npm run build` were re-run clean after every stage of this pass.

## RBAC round-trip elimination: JWT + in-memory TTL cache + permission versioning

A follow-on to the Performance pass above: `auth` (`UserRepository.get_by_id`'s `joinedload` of `User`+`Role`+`Category`) was left as the one fixed per-request Postgres round trip on every authenticated endpoint after the Interactions-tab fix collapsed everything else to one combined query. This phase eliminated it for the common case, without introducing Redis — see the root `CLAUDE.md`'s "Cross-service identity" section for the JWT-claim/cache-hit/cache-miss mechanics in full (`app/core/rbac_cache.py`, `app/dependencies/auth.py`, `User.permission_version`). Nothing in this frontend needed to change — the JWT contract only gained claims (backward compatible: a token missing them just always takes the DB path, same degrade-safe convention `permissions`/`scoped_permissions` already used), and every route's response shape is identical. Measured: `auth`-stage `Server-Timing` went from ~1.2–3.7s (cache miss) to ~0.1–0.3ms (cache hit) on the second and every subsequent request for the same session; SQL round trips per normal read request went from 2 to 1. Live-tested the actual security tradeoff, not just the happy path: deactivating a user via the RBAC admin UI leaves their already-cached session usable for up to the cache's TTL (30s default) before their next request is rejected — a deliberate, bounded-staleness tradeoff, not an oversight, and the only way to avoid a DB round trip on every single request without Redis-backed cross-process invalidation.

## Known regression and fix: a canceled request surfacing as a visible "canceled" error toast

The Performance pass above added `AbortController`-based stale-response guards to `InteractionsPage.tsx` and `useMailInbox.ts` (`fetchBaseTab`/`loadMoreBaseTab`), each checking `error.name === "CanceledError"` to silently swallow an expected cancellation (a newer request superseding an older one, or a page unmounting mid-request) rather than surfacing it as a failure. This silently didn't work: `api/client.ts`'s axios response interceptor unconditionally rewraps **every** rejected request into a plain `new Error(...)` before any caller's own `catch` sees it — for a canceled request (no `.response`), that lands on `error.message`, which happens to be the literal string `"canceled"`, but the rewrapped error's `.name` is just `"Error"`, not `"CanceledError"`, so the guard never matched. The cancellation fell through to `useMailInbox.ts`'s `refresh()` catch block (which toasts *any* caught error), producing a literal **"canceled"** toast on an entirely expected, harmless cancellation — reproduced by switching from the Interactions tab to Mail (or any similar rapid stale-request-superseding sequence). Fixed at the source: the interceptor now checks `axios.isCancel(error)` first and passes cancellations through unchanged, preserving their identity for every current and future caller instead of erasing it; the two guards were also switched from the fragile `.name` string check to `axios.isCancel(error)`, axios's own documented API, which stays correct regardless of how many layers an error passes through. **Any future code adding `AbortController` cancellation to a fetch that goes through `apiClient` must use `axios.isCancel(error)` to detect it, never `error.name`/`instanceof` checks** — this interceptor is why. The identical interceptor pattern exists in the standalone `ticketing-service/frontend`'s own `api/client.ts` (not fixed here, since that copy has no `AbortController` usage yet to expose the bug) — port this fix there first if that copy ever gains similar cancellation wiring. **`TicketsListPage.tsx` had this exact same unfixed bug** (`error.name === "CanceledError"`, line 117) — missed by the pass above despite living in this same file tree; a later audit (see the next section) found and fixed it the same way, `axios.isCancel(error)`. Don't assume every consumer of this pattern in a given commit actually got the fix — grep for `.name === "CanceledError"` (there should be none left) rather than trusting a summary like this one.

## Request-duplication audit: a persistent workspace mount, and the router desync it exposed

A follow-on performance investigation, prompted by Chrome DevTools captures showing 200-330+ requests per page load with heavy duplication of `GET /agents`, `/clients`, `/categories`, `/folders`, `/folder-counts`, `/view-counts` — refiring on *every* sidebar navigation, not just once per session as `WorkflowContext`'s own comment claims (see the Performance pass section above). Root cause, found by tracing the actual Next.js route structure rather than assuming the embedding was fine: `src/app/(dashboard)/dashboard/[[...slug]]/page.tsx` was an optional catch-all resolving every `/dashboard/*` URL to one page component, which mounted `<TicketWorkspaceApp/>` (and with it, `WorkflowProvider`, `ToastProvider`, and `react-router-dom`'s own `<BrowserRouter>`). The shell's sidebar (`components/layout/sidebar.tsx`) navigates via plain `next/link`, and **Next.js's App Router only preserves component state across a navigation for layouts, never for pages** — even when it's literally the same `page.tsx` resolving both the old and new URL, a slug change remounts it. So every click between Mail/Tickets/Interactions/Audit Logs tore down and rebuilt the whole embedded app, silently defeating `WorkflowContext`'s "once per session" fetch and every page hook's own "already loaded" cache (`useMailInbox`'s `loadedKeysRef`, `TicketsListPage`'s abort-guarded state, etc.) on every single navigation. `components/layout/dashboard-layout.tsx`'s `key={pathname}` (on the `motion.div` wrapping page content, for Framer Motion's enter/exit) compounded the same remount on the same trigger.

**Fix**: added `src/app/(dashboard)/dashboard/layout.tsx`, a new layout one segment above the catch-all page — Next.js *does* preserve a layout's state across navigation to any of its child routes, so mounting `TicketWorkspaceApp` there instead keeps it alive across every ticket-workspace-internal navigation. The layout carries the same role/slug-depth branching the catch-all page used to do itself (Viewer never sees the workspace; Super Admin/Site Lead/Account Manager/Team Lead/Staff see it only once the slug goes deeper than the bare `/dashboard` root, where they instead get their own role dashboard) — when it decides *not* to show the workspace, it renders `{children}`, i.e. the now-much-simpler `[[...slug]]/page.tsx`, which lost its own `TicketWorkspaceApp`-returning branches entirely (dead now that the layout gates them) and kept only the per-role root-dashboard logic. `dashboard-layout.tsx`'s animation key was changed from `pathname` to `pathname.split("/")[1]` (the top-level segment) — every ticket-workspace route shares the `"dashboard"` segment, so navigating between them no longer remounts the `motion.div` (and everything now persistently mounted inside it); a genuine top-level page change (e.g. `/dashboard` → `/users`) still gets a fresh key and its enter/exit animation.

**This surfaced a real, previously-invisible bug once the remount stopped masking it**: `react-router-dom`'s `<BrowserRouter>` only learns about a URL change through its own `push`/`replace` calls or a browser `popstate` event — it does not monkey-patch `window.history.pushState`, so Next's own client-side navigation (which does call the History API directly, just not through react-router's instance) changed the address bar and the shell's own highlighted nav item without react-router's `<Routes>` ever finding out, leaving the content pane frozen on whatever page was previously matched. Invisible before, because the old remount-per-navigation meant a fresh `BrowserRouter` always read the correct `window.location` at mount time. Fixed with a small bridge component, `RouterSync` (rendered inside `<BrowserRouter>` in `TicketWorkspaceApp.tsx`): it watches Next's own `usePathname()` and, whenever that changes, calls react-router's `navigate(relative, {replace: true})` to force its location to catch up — deliberately only reacting to Next's pathname changing, not to every react-router location change too, so it doesn't fight react-router's own in-app navigation (e.g. a page's own `useNavigate()` call opening a ticket row). Verified live via a headless Playwright run driving 9 real sidebar navigations across Mail/Tickets/Interactions/Audit Logs as an Account Manager: `GET /agents` fired exactly once for the whole session (previously once per navigation), `/clients`/`/categories` fired only during the initial load (2x each, consistent with React StrictMode's dev-only double-invoke — not a loop), every page rendered its correct role-scoped content after each navigation, and zero console errors. Page-specific content (the ticket list, mail counts, interactions, audit logs) still refetches on revisiting that specific page, which is correct/expected — the fix targeted the *global* reference-data duplication, not a page's own legitimate refresh-on-open.

**Smaller fixes from the same audit, all in the API layer**: `api/agent.ts`'s `listAgents(category)` (called directly by `TicketActions.tsx` and `MessageDetailsView.tsx` for a ticket's category-scoped agent picker, bypassing `WorkflowContext`'s unscoped cache entirely since it's a different, filtered query) now has a short-TTL (30s) in-flight cache keyed per category, so reopening the same ticket or category doesn't reissue an identical `GET /agents?category=...`. `api/sla.ts`'s `getTicketSla`/`listSlaPolicies` gained real `AbortSignal` support, threaded through `useTicketSla.ts`'s 8s poll, so a superseded/unmounted request is actually canceled at the network layer instead of only having its result discarded client-side after it completes on the wire. `api/auditLog.ts`'s `getTicketAuditLogs` now coalesces genuinely-concurrent calls for the same ticket id (clearing the moment the shared request settles, never holding a stale result) — `SlaTimeline.tsx` (always mounted) and `TicketAuditLog.tsx` (tab-gated, its own 10s poll) both independently call it for the same ticket, often at the same instant a ticket detail page first opens.

**Backend**: this same audit found the reported 27-48 second response times (e.g. on `GET /sla/policies`, a 3-row query with no way to genuinely take that long) were consistent with connection-pool queueing under the frontend duplication bug's 200-300 concurrent requests, not slow queries or missing indexes — the query/index/pagination work from the Performance pass above was confirmed still correct end-to-end. See the root `CLAUDE.md`'s "Cross-service identity" section for the resulting `unified-backend/app/database/session.py` pool-size increase and `rbac_cache.py` single-flight fix.

## SLA Overview tile: N+1 fix (`GET /tickets/sla-overview-counts`)

The Dashboard's "SLA Overview" tile row (Running/Paused/At Risk/Breached/Escalated/Completed) used to be computed entirely client-side by `useDashboardSlaCounts`: fetch every visible ticket unbounded (`listTickets()`), then call `GET /tickets/{id}/sla` once per ticket via `Promise.all`, showing `"…"` on every card until all of them resolved. Both `Dashboard.tsx` (this app's embedded ticket workspace) and the shell's own `SlaOverviewSection.tsx` (mounted inside `super-admin-dashboard.tsx`'s reused component — see "Per-role dashboards and Reports" above) independently triggered this by mounting the hook, each running its own full unbounded fetch. Measured directly against real dev data (an 85-ticket account): **~16.3 seconds** for the old pattern — this was the actual cause of a user-reported "SLA Overview shows 3 dots and takes forever to load" bug, not a rendering issue.

**Fix**: one grouped backend query, `TicketRepository.sla_overview_counts` — joins `tickets → resolution_slas → sla_policies` and classifies each `RUNNING` clock's tier via `EXTRACT(EPOCH FROM (due_at - now))` compared against the policy's target (mirroring `compute_elapsed_fraction`'s exact formula, expressed in SQL instead of Python, avoiding a division by encoding the same thresholds as remaining-seconds comparisons), under the same AM-owned-clients/TL-category visibility scoping as every other ticket endpoint. New route `GET /tickets/sla-overview-counts` (`SLAOverviewCountsResponse`). `useDashboardSlaCounts` was rewritten to call this one endpoint — it no longer takes a `tickets` argument at all, and both `Dashboard.tsx` and `SlaOverviewSection.tsx` dropped their own separate `listTickets()` fetch that used to exist purely to feed the hook.

**Verified**: the new endpoint returns in ~1.2–1.9s warm (the remaining time is per-request Neon/auth round-trip overhead already documented in the Performance-pass section above, not query execution — `EXPLAIN ANALYZE` on the equivalent query is sub-millisecond at current table sizes) vs. the old pattern's measured ~16.3s for the same account and data — over a 13x improvement. Its bucket counts were cross-checked byte-for-byte against a Python replication of the old client-side classification logic run against the same live data and matched exactly (`{running: 13, paused: 2, at_risk: 0, breached: 0, escalated: 11, completed: 2}` both ways) — same semantics preserved deliberately, not "fixed" into something new: `running` counts every `RUNNING` clock regardless of tier, and `at_risk`/`breached`/`escalated` remain non-exclusive tier sub-counts of that same set, exactly as the old client-side loop already counted them, so no existing consumer's numbers shift.

## Notification-driven deep links: a First Response SLA click now opens the specific message

Clicking a First Response SLA notification (from the topbar bell, or its mirror in the Mail "System" folder — see the root `CLAUDE.md`'s matching paragraph for the backend-side content enrichment and the `?interaction_id=` link this depends on) used to land on the bare `/inbox` with no way to tell which message triggered it. `InboxPage.tsx` now reads `interaction_id` via `useSearchParams()` (react-router-dom) on mount/param-change and calls the page's own pre-existing `handleOpen(id)` to open that specific thread.

**Making this work exposed two layered bugs in `RouterSync`** (`TicketWorkspaceApp.tsx`), the bridge component that keeps react-router's location in sync with Next's own client-side navigation (see "Request-duplication audit" above for why `RouterSync` exists at all) — both are now fixed and documented in the component's own comments, worth restating here since they're a subtle, generally-applicable React/Next.js gotcha:

1. **Query strings were silently dropped.** `RouterSync` originally watched only `usePathname()`, which excludes the query string entirely — a Next-driven navigation carrying `?interaction_id=...` synced react-router's location with just the bare path, so `InboxPage`'s own `useSearchParams()` had nothing to read even though the address bar showed the full URL correctly.
2. **The first fix attempt caused an infinite render loop.** Adding `useSearchParams()` (the object) directly to the syncing effect's dependency array seemed like the obvious fix, but `next/navigation`'s `useSearchParams()` returns a **new** `ReadonlyURLSearchParams` instance on every render even when its content is unchanged — so the effect never stabilized, each firing called `navigate()`, which triggered another render, which produced another new object instance, forever. Confirmed via temporary injected `console.log` output showing hundreds of identical firings within ~2 seconds, with the actual route never advancing past the pre-navigation URL despite the address bar showing the correct target. Fixed by depending on the **string** `useSearchParams().toString()` instead of the raw object — a stable primitive that only changes when the query actually does.

Separately, `resolveNotificationHref()` (`top-navbar.tsx`) had a narrower version of the same bug: its `TICKET_WORKSPACE_ROUTES` prefix match only checked `link === route || link.startsWith(\`${route}/\`)`, missing the case where a route is immediately followed by a `?query` with no `/` in between — fixed by adding `|| link.startsWith(\`${route}?\`)`. Verified live end-to-end (not just via direct URL, which had earlier masked the `RouterSync` bugs entirely): a real notification bell click and a real Mail-System-folder click both now open the specific message, not the inbox.
