# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Enterprise RBAC (Role-Based Access Control) User Management System — a FastAPI backend + Next.js frontend monorepo. Six roles manage users, roles, and permissions across a reporting hierarchy (`manager_id` / `teamlead_id` on each user): **Super Admin** (rank 5, all permissions) > **Site Lead** (rank 4, all permissions except `ticket:system_config`/`audit:export`) > **Account Manager** (rank 3, renamed in-place from "Manager" — same `role_id`, no orphaned data) > **Team Lead** (rank 2) > **Staff** (rank 1), plus **Viewer** which sits outside the rank ladder as a client-facing role with a small fixed permission set (`user:view`, `role:view`, `permission:view`). Site Lead and Viewer are both deliberately excluded from `organization_service.py`'s `ROLE_HIERARCHY`/`_build_subtree` (which only spans Super Admin → Account Manager → Team Lead → Staff) — Site Lead has unconditional visibility of everything rather than a place in the reporting tree.

**This directory is `unified-frontend/`** (formerly `rbac-service/frontend/`, one level down inside a since-retired `rbac-service/` parent — see the root `CLAUDE.md`'s frontend-consolidation note). Next.js 16 (App Router) + TypeScript + Tailwind + Shadcn UI. Its FastAPI backend counterpart lives at the monorepo-root `unified-backend/app/rbac/` (see "Commands" below and the root `CLAUDE.md`'s "Backend consolidation" section) — the two used to be sibling `backend/`/`frontend/` subdirectories under one `rbac-service/`, now they're two independent top-level directories with no shared parent.

**Path note for the rest of this document**: everywhere below that says `frontend/...` (a holdover from when this file lived one level up, in `rbac-service/`) now means "relative to this directory" — e.g. `frontend/src/lib/api.ts` is just `src/lib/api.ts` from here. Not rewritten line-by-line throughout; translate mentally.

This service (this frontend, plus the `app/rbac/` half of `unified-backend`) is the **shell application**: it owns authentication, session, role-based routing/navigation, and layout for the whole product. It is also the **sole issuer** of the JWT access/refresh tokens `app/ticketing` verifies — now a module boundary inside one process rather than two separate ones, see root `CLAUDE.md`. Rotating `JWT_SECRET_KEY` in `unified-backend/.env` invalidates every session across both domains.

**Product branding is "UTMS" / "Unified Ticket Management System"**, not "RBAC" / "Enterprise Platform" — the sidebar (`components/layout/sidebar.tsx`), the mobile header (`components/layout/dashboard-layout.tsx`), and the `<title>` metadata (`app/layout.tsx`) were all renamed from the original RBAC-only branding once the product became the combined ticketing+RBAC system. "RBAC" still appears throughout this document and in code comments/identifiers (role names, internal module names, etc.) — that's the system's technical/historical name, not a stale branding reference; only the **user-facing** product name changed.

**The Ticket Management workspace is embedded here, not linked to.** `ticketing-service/frontend` was copied (not just referenced) into `src/ticket-workspace/` and mounted as a client-only `react-router-dom` subtree inside this Next.js app — see "Ticket workspace embedding" under Frontend architecture below. Staff, Team Lead, and Manager land there after login; Super Admin and Viewer instead see this app's own admin dashboard. The two frontends are **not** kept in sync automatically — a ticket-related UI change usually needs to be made in both places (`ticketing-service/frontend/...` and this directory's own `src/ticket-workspace/...`, same relative structure).


## Architecture

**Path note**: every `app/...` path below is relative to `unified-backend/app/rbac/`, not this repo's own (now-empty) `backend/`. `unified-backend/alembic_rbac/` is this domain's migration chain (not `backend/alembic/`).

### Backend: layered, but authorization stops at the frontend

Routes (`app/api/v1/`) → Services (`app/services/`) → Repositories (`app/repositories/`) → SQLAlchemy models. Two model sources are merged into one `Base.metadata`: `User` and `Role` come from `shared_models` (a local editable install both `app/rbac` and `app/ticketing` import — see root `CLAUDE.md`), while `Permission`, `RolePermission`, `AuditLog`, `UserPermissionOverride`, and `PermissionRequest` are local (`app/rbac/models/`). `unified-backend/alembic_rbac/env.py` (formerly `backend/alembic/env.py`) imports both sources and filters out `app/ticketing`'s tables via `include_object` so `alembic revision --autogenerate -c alembic_rbac/alembic.ini` only diffs this domain's own tables.

**Every route only checks authentication (`Depends(get_current_active_user)`), never authorization — with one deliberate exception (see below).** There is no generic `require_permission`-style dependency anywhere in the backend — any authenticated user can call almost any endpoint (list all users, change any role's permissions, etc.) regardless of their role or permission set. Everything else — permission-gated buttons, role-based page visibility, the Users page's reporting-hierarchy filtering — is still enforced **only in the frontend**. Treat this as a known architectural constraint, not something to silently "fix" by adding backend checks unless asked.

**The one exception is per-user permission overrides.** `POST/GET/DELETE /users/{id}/permission-overrides` (`api/v1/permission_overrides.py`) is the first real server-side authorization check in this backend — see "Per-user permission overrides" below for the full shape. Its check, `PermissionOverrideService.ensure_can_manage_overrides` (public — not prefixed with `_`, precisely because a second caller, `PermissionRequestService`, now reuses it directly rather than re-deriving the same role-scoping rules; see "Permission requests" below), was deliberately kept as a direct service-layer method rather than a generic dependency until there's a third real call site.

Audit logging is defined (`AuditLogService`, `audit_logs` table, `GET /audit-logs`) and **is now actually called** — by the permission-override grant/revoke flow (`PermissionOverrideService`) and the permission-request create/approve/reject flow (`PermissionRequestService`), which together are still its only writers. Every other user/role/permission mutation in this backend still leaves the table empty; don't assume a row exists for anything else without checking.

**Seeding is not purely additive** (`scripts/rbac_seed/seed.py`): the main loop only ever *adds* a permission grant a role is missing per `DEFAULT_ROLES` — it never revokes one just because a role's default list changed. Two explicit cleanup lists run once near the top of `seed()`, before the main loop, to handle everything the additive loop can't: `DEPRECATED_PERMISSIONS` (permission names to delete outright, including their `role_permissions` grants — e.g. `ticket:bulk_reassign`, `ticket:configure_routing`) and `REVOKED_GRANTS` (specific `(role_name, permission_name)` pairs to un-grant without deleting the permission itself — e.g. Staff's `ticket:create`). When narrowing a role's permissions going forward, add the removal to one of these two lists — editing `DEFAULT_ROLES` alone will not revoke anything from a DB that already has the old grant. There's also a one-time in-place role rename block (`Role.name` mutated directly, same `role_id`, guarded so it only fires if the old name exists and the new one doesn't yet) — this is the safe pattern for renaming a role without orphaning users/grants, used for the "Manager" → "Account Manager" rename.

**Database URL normalization** (`app/core/config.py`, `Settings.normalize_database_url`) is important to understand before touching `DATABASE_URL` handling: it rewrites `postgres://`/`postgresql://` → `postgresql+asyncpg://`, translates `sslmode=` → `ssl=`, and strips `channel_binding=` — all required because Neon's connection strings use kwargs asyncpg doesn't accept. `alembic/env.py` does the reverse translation (`+asyncpg` stripped, `ssl=` → `sslmode=`) to get a psycopg2-compatible sync URL for migrations. Don't naive-string-replace this logic — use `urllib.parse`, per the existing implementation.

### Per-user permission overrides

A role's permission bundle (`role_permissions`) is the default; `user_permission_overrides` (`app/models/permission_override.py`) is a small, purely **additive** exception list layered on top — granting one specific person a capability their role doesn't include, without touching the role or affecting anyone else who holds it. Soft-revocable (`revoked_at`/`revoked_by`, never hard-deleted) with an optional `expires_at` (blank = permanent), and a partial unique index (`WHERE revoked_at IS NULL`) so at most one *active* grant exists per user+permission at a time while a full grant→revoke→re-grant history is preserved.

**Overrides can optionally be scoped to one specific ticket** (`scope_ticket_id`, nullable UUID, no FK — see below for why). `NULL` (the common case, and the only kind this section originally described) means the grant applies everywhere the permission would normally apply; a non-`NULL` value means it applies *only* for that one ticket. This exists for exactly one workflow today: a Staff member requesting `ticket:editother_ticket` (see "Permission requests" below) scoped to one specific teammate's ticket, rather than a blanket grant to work every ticket in scope. The active-grant uniqueness index is `(user_id, permission_id, COALESCE(scope_ticket_id, <sentinel-uuid>)) WHERE revoked_at IS NULL` — the `COALESCE` matters: a plain 3-column unique index would treat every `NULL` as distinct from every other `NULL` (Postgres's default null-handling) and silently allow two simultaneous *global* grants for the same user+permission, which isn't what "at most one active global grant" should mean. `scope_ticket_id` is deliberately a bare UUID with no `ForeignKey` — `tickets` is owned by `app/ticketing`'s own migration chain (`alembic_ticketing`), and a cross-chain FK would tie the two histories together the same way the root CLAUDE.md's "never generate a migration in one chain that touches the other's tables" rule already warns against; validity is checked in application code instead (`PermissionRequestService.create_request` looks the ticket up via `app.ticketing`'s own `TicketRepository`, imported directly since both domains are one process now).

**`PermissionResolverService`** (`app/services/permission_resolver.py`) is the single place effective permissions are computed: `role_permissions(role_id) ∪ active_global_overrides(user_id)` for the flat list, plus a **separate** `scoped_permissions: dict[str, list[str]]` return value (permission name → ticket ids) built from active *ticket-scoped* overrides only. This split matters: a ticket-scoped override is deliberately excluded from the flat permission list, so a plain `has_permission(user, "ticket:editother_ticket")` membership check can never mistake "holds it for one ticket" for "holds it everywhere" — call sites that need to check a scoped grant use `app.ticketing.services.access_control.has_permission_for_ticket(user, permission_name, ticket_id)` instead, which checks both the flat list and the scoped map. `AuthService.get_current_user`/`login`/`refresh_token` all call `get_effective_permissions` and thread all three return values through — `permissions` and `scoped_permissions` both end up as separate JWT claims (see root CLAUDE.md's "Cross-service identity"), and `override_permissions` additionally appears on `/auth/me`'s `CurrentUser` response, alongside `scoped_permissions` itself (both nullable/optional on the frontend's `AuthUser` type, `src/types/index.ts`, and passed through as-is by `authService.me()`, `src/services/index.ts`). `ticketing-service`'s `TicketActions.tsx` reads `currentUser.scoped_permissions` to disable-in-place ticket actions the caller can't perform on that specific ticket — see that repo's CLAUDE.md. Never call `RolePermissionRepository.get_permissions_by_role` directly to answer "what can this user do" — that only tells you what their role grants, not their actual effective set.

**Grant/revoke authorization** (`PermissionOverrideService.ensure_can_manage_overrides`) is scoped by the actor's own role: Super Admin/Site Lead have unconditional authority; Account Manager is restricted to their own reports, resolved via `OrganizationService.get_subordinate_user_ids` (flattens the existing `_build_subtree` org-chart tree rather than duplicating the `manager_id`/`teamlead_id` traversal). `grant()` rejects a permission the target's role already includes (400 — "would be redundant") — this check is unaffected by scoping, since a ticket-scoped grant is for a permission the target's role does *not* hold at all (e.g. Staff and `ticket:editother_ticket`), never a permission they already have generally. Every grant/revoke writes a row via `AuditLogService.create_log` — see the audit-logging note above.

**Frontend**: `components/users/user-permission-overrides.tsx` renders a "Personal Permission Grants — {name} only" section inside `user-detail-drawer.tsx`, **below and visually distinct from** the existing role-permissions editor — the two are unrelated mutations (one edits the role everyone with it shares, the other edits one person) that were previously easy to conflate since only the role editor existed. Gated by `<PermissionGuard permission="permission:override_grant">`, same convention as everywhere else. The grant form filters out permissions already in the target's role bundle client-side (mirroring the backend's redundant-grant rejection) so the UI never offers a choice the API would reject.

### Permission requests

A self-service layer on top of the override mechanism above: `permission_requests` (`app/models/permission_request.py`) lets any user ask for a permission they don't currently have, addressed to a *role* (e.g. "Account Manager") rather than a specific person — any user holding that role can review it, subject to the exact same authority `ensure_can_manage_overrides` already enforces (Super Admin/Site Lead see and can act on every pending request regardless of which role it's addressed to; an Account Manager only sees/acts on ones from their own reports). This is deliberately a request/review *workflow* layered on top of, not a reimplementation of, the override mechanism — approving one calls `PermissionOverrideService.grant()` directly and stores the resulting `override_id` back on the request row (`granted_override_id`), so there is exactly one code path that ever creates a `UserPermissionOverride`, and a request's "is this now revoked" state is read through that link rather than duplicated.

**`PermissionRequestService.list_eligible_approver_roles`** answers "which roles could even grant this permission" by querying `role_permissions` **live** for every role (never `seed.py`'s static `DEFAULT_ROLES`, which can drift from the DB — see the seeding note above) and keeping only roles that hold both `permission:override_grant` and the requested permission itself — a role can't vouch for a permission it doesn't hold. This is also why Team Lead never appears as an approver option today: it holds neither, regardless of what CLAUDE.md or seed.py comments might suggest a future role matrix could look like.

**A request can optionally carry `scope_ticket_id`**, turning it into a ticket-scoped grant instead of a blanket one (see "Per-user permission overrides" above) — `approve()` passes it straight through to `PermissionOverrideService.grant()` unchanged, so there's still exactly one place a `UserPermissionOverride` gets created. Two purpose-built read-only endpoints back the request form's dynamic fields, both under `/permission-requests/scope/`:
- `GET .../staff-options` — the requester's teammates (other Staff sharing the same `teamlead_id`, via `UserRepository.get_by_teamlead` — the same lookup `OrganizationService._build_subtree` already uses for a Team Lead's org-chart children, reused rather than reimplemented).
- `GET .../ticket-options?staff_id=` — that teammate's assigned tickets, via `app.ticketing`'s own `TicketRepository.list_all(agent_id=staff_id)` filtered to `agent_id == staff_id` exactly (the repository method's own `agent_id=` branch deliberately also includes unassigned tickets for its original shared-pool-visibility caller, so the service layer re-filters). `staff_id` is validated as a genuine teammate first (`PermissionRequestService._is_teammate`) — this endpoint intentionally bypasses the normal category-scoped ticket-visibility rule (`ensure_agent_can_view_ticket`) that a regular ticket list would apply, since the actual security boundary here is "is this really your teammate," not category; don't reuse it as a general-purpose ticket-listing endpoint.

Both fields only ever appear in the frontend when the selected permission is `ticket:editother_ticket` — and since `eligiblePermissions` already excludes permissions the requester already effectively holds, and `editother_ticket` is a role default for everyone except Staff, in practice only Staff members ever see these fields at all (no separate role check needed to gate them).

**Frontend**: a full page at `/permission-requests` (`app/(dashboard)/permission-requests/page.tsx`) with two tabs — "My Requests" (visible to everyone) and "Pending My Review" (gated by `<PermissionGuard permission="permission:override_grant">`, hidden entirely otherwise). Reachable from **every** role's sidebar, including Staff/Team Lead/Account Manager — see the role-access.ts note below for why that's not the architectural fork it might look like. No push-notification system was built for this (the topbar bell is still the pre-existing UI-only mock, see below) — "notification" is satisfied by the pull-based "My Requests" tab reflecting the decision once you reload it, deliberately mirroring how `ticketing-service`'s Edit Access workflow already works.

### Frontend: App Router + centralized role-access config

- **Ticket workspace embedding**: `src/ticket-workspace/` is an adapted copy of `ticketing-service/frontend/src/`. `tsconfig.json` maps `@tw/*` → `./src/ticket-workspace/*` (separate from this app's own `@/*` → `./src/*`) so the two codebases' imports can never accidentally cross. `src/ticket-workspace/TicketWorkspaceApp.tsx` mounts the copied page tree under a `react-router-dom` `<BrowserRouter basename="/dashboard">`; `src/app/(dashboard)/dashboard/[[...slug]]/page.tsx` is the optional-catch-all Next.js route that renders it (via `next/dynamic(..., { ssr: false })` — `BrowserRouter` reads `window.location` at render time, which crashes Next's one server-render pass otherwise) for Staff/Team Lead/Account Manager (there is no role literally named `"Manager"` — see `ticketing-service/CLAUDE.md`'s "Roles" section if you see that string hardcoded anywhere, it's dead), `src/components/dashboard/site-lead-dashboard.tsx` for Site Lead (its own dedicated dashboard, not routed into the ticket workspace — Site Lead's day-to-day is org oversight, not hands-on ticket work), or `src/components/dashboard/viewer-dashboard.tsx` (this app's own original admin dashboard, pre-embedding) for Viewer and Super Admin. `src/ticket-workspace/context/AuthContext.tsx` is a thin adapter re-exposing this app's own `useAuthStore` instead of holding its own session state — the embedded copy has no login/token logic of its own; auth is exclusively this shell app's responsibility. `src/ticket-workspace/components/layout/AppLayout.tsx` similarly only renders a per-page title bar via this app's own `PageHeader` — Sidebar/Topbar/AuthGuard all come from this app's own `DashboardLayout`, one level up.
- **Design-token unification**: `.tm-scope` (`src/app/globals.css`) remaps the ticket workspace's own CSS variables (canvas/surface/accent/warning/danger/success/info/teal, plus the `slate`/`border`/`muted` scale) onto this app's actual computed dark-theme values (converted once from its HSL shadcn tokens) so the workspace and this app's own pages (Profile/Settings/Users/Roles) render as one consistent neutral-dark design system, not two visually distinct products. This applies to every role that reaches the workspace (Staff/Team Lead/Manager) uniformly — there is no per-role theme branching. Extending Tailwind's `content` glob to include `src/ticket-workspace/**/*` in `tailwind.config.ts`, plus a handful of non-colliding token additions there, is what makes the copied components' utility classes compile at all.
- `src/lib/role-access.ts` is the single source of truth for role-based UI behavior: `ROLE_NAMES` constants, per-role sidebar visibility (`NAV_ITEMS_BY_ROLE`, including which roles see the ticket-workspace nav items vs. this app's own Users/Roles vs. both — e.g. "Create Dummy Mail" is included for `ACCOUNT_MANAGER` but not `TEAM_LEAD`/`STAFF`), `SUPERVISOR_ROLE_NAMES`/`isSupervisorRole()`, and which roles a given role is allowed to assign on user creation (`CREATABLE_ROLES_BY_ROLE`). **`SUPERVISOR_ROLE_NAMES` here is only `[SITE_LEAD, SUPER_ADMIN]` and does *not* actually mirror `ticketing-service`'s real backend role set** (`{"Team Lead", "Account Manager", "Site Lead", "Super Admin"}`, in that repo's `access_control.py`) — it happens to match `ticketing-service/frontend`'s own same-named-but-equally-narrow `SUPERVISOR_ROLES` constant, but neither frontend constant matches the backend's actual set. This is known, pre-existing drift, not yet reconciled — don't assume any two of these three role-sets agree without re-reading all three. New role-conditional UI logic should extend this file rather than hardcoding role-name strings in components. Note "Ticket Audit Log" (the ticket workspace's own per-ticket audit trail nav item) is a deliberately distinct `NavItemKey` from this app's own "Audit Logs" (the user/role/permission audit page) — same underlying concept, different systems, kept apart to avoid a naming collision in the sidebar.
- **`NAV_ITEMS_BY_ROLE` is one shared sidebar config for every role, not two separate sidebars.** It's easy to misread "Staff/Team Lead/Account Manager land on the ticket workspace after login" (above) as meaning those roles never see this app's own RBAC-native pages, but `src/components/layout/sidebar.tsx`'s `SidebarContent` renders from the exact same `menuItems`/`NAV_ITEMS_BY_ROLE` regardless of role — it just mixes ticket-workspace routes (`/dashboard/inbox`, `/dashboard/tickets`, ...) and RBAC-native routes (`/users`, `/roles`, `/permission-requests`, ...) in one list, filtered per role by `canSeeNavItem`. So a *new* RBAC-native page only needs one addition here (`NavItemKey` + `NAV_ITEM_TRANSLATION_KEY` + a `menuItems` entry in `sidebar.tsx`) to become reachable from every role's sidebar, including the ones that land in the embedded ticket workspace by default — no separate ticket-workspace-embedded page or cross-service API call is needed just to reach it. `Permission Requests` (see above) is the first page built this way.
- **The topbar notification bell (`top-navbar.tsx`) is a UI-only mock** — a hardcoded in-memory array, no backend model, no `GET /notifications` endpoint. Don't assume it reflects anything real (pending permission requests, override expirations, etc.) unless a future task explicitly wires it to a real data source.
- **Permissions include a Ticket Management group**: `unified-backend/scripts/rbac_seed/seed.py`'s `DEFAULT_PERMISSIONS` includes the `ticket:*`/`communication:*` permission catalog (create, view_own, view_others, assign, transfer, change_priority, change_category, escalate, `editown_ticket`, `editother_ticket`, etc.) assigned per-role to mirror the ticket workspace's actual capability matrix — `ticket:bulk_reassign`, `ticket:configure_routing`, and (as of the `editown_ticket`/`editother_ticket` split) the old single `ticket:edit_ticket` were all removed outright (see `DEPRECATED_PERMISSIONS` in `seed.py`, which deletes the rows and their grants rather than leaving them orphaned); don't reintroduce any of the three without checking whether that removal reasoning still applies. `src/components/users/user-detail-drawer.tsx` renders them as a "Ticket Management" card via the same prefix-based grouping (`permission_name.split(":")[0]`) used for `user:`/`role:`/`audit:` permissions. **These used to be entirely self-contained bookkeeping** (`app.ticketing` authorized purely by role name, not these rows) **but are now partially real**: the JWT's `permissions`/`scoped_permissions` claims (see the root CLAUDE.md's "Cross-service identity" section) propagate a user's effective permission set to `app.ticketing`, which enforces a growing subset of them server-side via `ensure_has_permission`/`has_permission_for_ticket` — currently `ticket:change_priority`, `ticket:transfer` (Staff-only, layered on top of the pre-existing role-based supervisor check), and the `ticket:editown_ticket`/`ticket:editother_ticket` pair (see `ticketing-service/CLAUDE.md`'s "Permission-based enforcement" and "Edit access requests" sections). Most of the catalog is still unenforced bookkeeping — grep `unified-backend/app/ticketing/services/access_control.py` for the current, real list of `ensure_has_permission`/`has_permission_for_ticket` call sites before assuming any given `ticket:*` permission actually gates anything.
- `src/components/auth/AuthGuard.tsx` gates the whole `(dashboard)` route group on `authService.me()` resolving; `PermissionGuard` conditionally renders based on `useAuthStore().hasPermission()`. Individual pages (e.g. Users) additionally self-gate with a role allowlist check that renders `AccessDenied` (403) for disallowed roles — this is a page-level pattern, not a route middleware, since `middleware.ts` only handles public/auth routing, not per-role authorization.
- Users page hierarchy filtering is entirely client-side: fetch all users (`page_size: 100`), then filter in-memory by `manager_id`/`teamlead_id` against the current user, since the backend has no server-side filtering for this and no row-level authorization (see above).
- Permission editing lives in `src/components/users/user-detail-drawer.tsx` (a right-side `Sheet`), not a standalone page — there is no `/permissions` route. Opening a user's drawer shows **two separate** permission-editing surfaces: the role-permissions editor (`PUT /roles/{role_id}/permissions`, edits **every** user who holds that role) and, below it, the per-user "Personal Permission Grants" section (`user-permission-overrides.tsx`, edits only this one person — see "Per-user permission overrides" above). A Manager editing role permissions can only toggle permissions they personally hold (checked against their own `AuthUser.permissions`); Super Admin is unrestricted; Team Lead/Staff/Viewer can't reach either surface at all.
- i18n is a small custom system, not a library: `src/lib/i18n/translations.ts` (flat dictionary keyed by dotted strings, one object per language) + `src/hooks/use-translation.ts` (reads the active language from `useSettingsStore`, which persists via zustand). Adding a UI string requires adding the same key to every language dictionary — `Record<keyof typeof en, string>` on the non-English dictionaries enforces this at compile time.
- `src/components/shared/data-table.tsx` (`DataTable`/`DataTablePagination`) wraps `@tanstack/react-table` and is reused by both Users and Audit Logs; it supports an optional `onRowClick` for row-level navigation/drawers without needing per-page reimplementation.
- Zustand stores (`src/store/`) are the persistence layer for anything not backed by the API: `auth-store.ts` (current user + theme), `settings-store.ts` (language, notifications, security prefs, mock session list), `profile-extras-store.ts` (phone/address/avatar URL — fields with no backend column, stored client-side only and not synced across devices).
- Backend API base URL is `process.env.NEXT_PUBLIC_API_URL` (`src/lib/api.ts`), falling back to `http://localhost:8000/api/v1` for local dev. Being a `NEXT_PUBLIC_*` var, it's baked in at build time — changing it in a deployed environment requires a rebuild, not just a restart. The embedded ticket workspace's **separate** axios instance (`src/ticket-workspace/api/client.ts`) uses its own `NEXT_PUBLIC_TICKETING_API_URL`, falling back to `http://localhost:8001` — a stale default from before the backend merge, see the matching Known Issues entry below.

## Known Issues

- `npm run lint` is broken: Next.js 16 dropped `next lint`, but the project still has an old-format `.eslintrc.json` rather than an ESLint 9 flat config. Use `npx tsc --noEmit` to check for real errors instead.
- `docs/ARCHITECTURE.md`, `docs/API.md`, and `docs/DEPLOYMENT.md` describe an earlier/aspirational design that has drifted from the actual implementation (e.g. they document a `refresh_tokens` table, `require_permission()` backend enforcement, and extra `/users` query params like `role_id`/`is_active` — none of which exist in the current code). Don't treat `docs/` as authoritative; verify against the actual source.
- **Turbopack workspace-root inference**: without the `turbopack.root` pin in `next.config.mjs`, Next.js 16's Turbopack walks up the filesystem looking for a lockfile to infer the project root and can land on an unrelated one several directories up (e.g. a stray `package-lock.json` in a parent/home directory) — this 404s every single route with no obvious error. `next.config.mjs` already pins `turbopack.root` to `__dirname` to prevent this; if every route suddenly 404s again after a config change, verify that pin is still intact before debugging anything else. Separately, a `next dev` process that's been running across many long sessions (especially through `npm install`/`uninstall` churn in the same directory, e.g. temporarily installing Playwright for verification) can land in a similar all-routes-404 state even with the config correct — a clean restart (kill the process, `npm run dev` again) resolves it; this is a stale-process issue, not a config regression, and is worth ruling out first.
- **`get_settings()` is `@lru_cache`d** (`backend/app/core/config.py`) — `.env` changes while the backend is already running have no effect until the process is restarted (`uvicorn --reload` only restarts on Python file changes, not `.env` edits). A config-dependent error that persists after editing `.env` almost always means "restart the server," not "the code is wrong." Same gotcha applies to `ticketing-service`'s backend.
- **A stale backend process can look exactly like "you have no eligible data" instead of an error.** A brand-new route added to a running `uvicorn` process that never picked up the change (see the `--reload` gotcha above) 404s; if the calling frontend code doesn't distinguish `isError` from "the query succeeded and genuinely returned an empty list," a 404 renders as a plausible-looking empty state instead of a visible failure — this happened for real with the Permission Requests dialog (rendered "You already have every permission" when the backend was simply missing the route). Any new React Query-backed dropdown/list should branch on `query.isError` explicitly, not just `query.isLoading` vs. "array is empty" — and if a brand-new page's data looks suspiciously, uniformly empty right after adding backend routes, check `GET /openapi.json` against the port the frontend is actually calling before assuming the frontend or the business logic is wrong.
- **The embedded ticket workspace's stale `:8001` default silently network-errors every ticketing request while RBAC's own requests keep working.** After the backend merge (see root CLAUDE.md), nothing listens on port 8001 anymore — everything is one process on 8000. `src/ticket-workspace/api/client.ts`'s `NEXT_PUBLIC_TICKETING_API_URL || "http://localhost:8001"` fallback was never updated, so an unset env var means every `tickets`/`inbox`/`interactions` call fails with a browser network error (visible in DevTools as `(failed) net::ERR_CONNECTION_REFUSED` or similar) while `notifications`/`users`/`auth` calls succeed — a confusing, asymmetric failure that looks like a backend bug but is a frontend config issue. Fix: add `NEXT_PUBLIC_TICKETING_API_URL=http://localhost:8000` (root, no `/api/v1` — see "Backend consolidation" in root CLAUDE.md for why ticketing routes are unprefixed) to `.env.local` (this directory's own), then restart `npm run dev` (`NEXT_PUBLIC_*` vars are baked in at server start, not hot-reloaded).
- **A dependency declared in `package.json` isn't necessarily installed** — the Mail rich-text composer (`src/ticket-workspace/components/mail/RichTextEditor.tsx`, `@tiptap/*` packages) was added to `package.json`'s `dependencies` without `npm install` ever being re-run afterward here, so `node_modules/@tiptap` was simply missing. Next.js's resulting error (`Module not found: Can't resolve '@tiptap/react'`, repeated for every `@tiptap/*` import, plus a same-shaped 500 on every route that imports it transitively) reads like a real code/config bug, not a "you forgot to install" one. If a module-not-found error names a package that's clearly already used elsewhere in the file/repo, check `node_modules/<package>` before assuming the import path or the code is wrong — `npm install` (no arguments needed, it reads `package.json`) resolves it, then restart `npm run dev` (a newly-installed package isn't picked up by hot-reload).
- **Turbopack's persistent `.next/` cache is not relocatable across a directory move/rename.** When this directory was renamed from `rbac-service/frontend` to `unified-frontend` (see the top-of-document note), the `.next/` build cache moved along with it — but Turbopack's cache stores absolute paths tied to the *old* location, so `npm run dev` afterward panics repeatedly with `FATAL: An unexpected Turbopack error occurred` / `Next.js package not found` (a `next-panic-*.log` gets written to `%TEMP%` each time). The `turbopack.root` pin in `next.config.mjs` does **not** cover this — that only fixes workspace-root *inference*, not a stale cache's baked-in paths. Fix: stop the dev server, `rm -rf .next`, restart `npm run dev` — do this any time this directory (or its parent) is moved/renamed again, before debugging anything else.
- **A committed `.env` file and two committed `*.tsbuildinfo` build-cache files were found during a repo-wide audit — in `ticketing-service/frontend`, not this directory** (this directory's own `.gitignore` already correctly excludes both patterns). See that repo's `CLAUDE.md` for the specific files; not yet cleaned up as of this note.
- **`src/ticket-workspace/api/notifications.ts` is dead code in this copy** — zero importers anywhere in `src/ticket-workspace/` (confirmed by grep). Its counterpart in the standalone `ticketing-service/frontend` app *is* used (by that app's own `Topbar.tsx`), but the embedded copy relies on the shell's own top-navbar bell instead (itself a UI-only mock, see above) — this file is leftover from the initial whole-tree copy, never wired up here, and not currently imported by anything. Safe to delete unless a future task actually needs it.
- **`NEXT_PUBLIC_TICKETING_FRONTEND_URL` is a dead env var** — documented in `.env.example` and `render.yaml` (as backing a "sidebar 'Ticket Workspace' opens in a new tab" link), but no component anywhere in this codebase actually reads it; `components/layout/sidebar.tsx` routes to the ticket workspace via plain internal Next.js links instead, confirmed by grep. Don't assume setting it does anything.

## Service Unification (RBAC + Ticket Management)

A separate, larger phase merged this originally-standalone RBAC product with a previously-independent Ticket Management service into one product:

- **Repo consolidation**: three independent git repos (`rbac-project`, `Phase_1`/ticketing, `shared_models`) merged into this one monorepo via `git subtree add` (full history preserved, `git log --follow` still works) as `rbac-service/`, `ticketing-service/`, and `shared_models/`.
- **Cross-service JWT trust**: this service became the sole token issuer; `ticketing-service`'s backend became a verify-only consumer sharing `JWT_SECRET_KEY`. `ticketing-service`'s old unauthenticated `agent_name` mechanism was removed entirely — see that repo's CLAUDE.md for the full shape.
- **Ticket Management workspace embedding**: `ticketing-service/frontend` copied into `src/ticket-workspace/` and mounted inside this Next.js app (see "Ticket workspace embedding" above) so the two products feel like one, rather than two apps cross-linked by an external URL (an earlier, since-superseded integration step).
- **Design-system unification**: the ticket workspace's own multi-hue palette was remapped onto this app's real dark-theme tokens (canvas/surface/text/border included, not just accent colors) so there's no visible seam between ticket pages and this app's own pages.

## Per-role dashboards and Reports (Account Manager / Team Lead / Staff)

Account Manager, Team Lead, and Staff now land on **the same dashboard layout as Super Admin** at the bare `/dashboard` root (previously they had no RBAC-native dashboard at all — they fell straight through to `TicketWorkspaceApp` regardless of slug; any deeper slug like `/dashboard/inbox` still does). `app/(dashboard)/dashboard/[[...slug]]/page.tsx` gained three more `role === ... && slug.length === 0` branches alongside the existing Super Admin/Site Lead/Viewer ones.

**Reuse, not duplication**: `components/dashboard/super-admin-dashboard.tsx`'s `SuperAdminDashboard` took two new optional props, `tickets?: MockTicket[]` (defaults to `MOCK_TICKETS`) and `description?: string` — every KPI/chart/list inside it was already a pure derivation of a `tickets` array, so making it reusable needed no other change, and Super Admin's own rendering is byte-identical to before. Three thin wrapper components — `account-manager-dashboard.tsx`, `team-lead-dashboard.tsx`, `staff-dashboard.tsx` — each read `useAuthStore`, compute a role-scoped ticket subset, and render `<SuperAdminDashboard tickets={...} description="..."/>`. The Reports page (`app/(dashboard)/reports/page.tsx`) applies the identical scoping to the same `useMockTicketsStore` data before feeding it into `getReportMetrics`/`getCountsByXxx`, so CSV/Excel/PDF exports are scoped too; "Reports" was added to `NAV_ITEMS_BY_ROLE` (`lib/role-access.ts`) for all three roles (previously Super Admin/Site Lead only) — the page itself still has no role-based access gate of its own (unlike `/users`), so reachability is nav-visibility only, not enforced.

**The scoping helpers are a deliberate workaround, not real ownership data — this is worth understanding before extending it.** `lib/mock-tickets.ts` gained `getTicketsForAccountManager(userId, tickets?)`, `getTicketsForTeamLead(userId, tickets?)`, and `getTicketsForStaff(userId, tickets?)`. `MockTicket` has no id-based owner field at all (only free-text `assignedTo`/`client` names drawn from this file's own fictional roster — `AGENTS`/`CLIENTS`), and none of those fictional names ever match a real seeded/created RBAC user's actual name. Matching by literal name (the obvious first approach) would show an **empty** dashboard for every real account. Instead, each helper deterministically buckets a real user — by hashing their own `user_id` (`stableIndex`, a simple non-random hash) — onto one fixed slice of the fictional data: Staff gets one fictional agent's tickets, Team Lead gets one of the three `AGENT_TEAM` teams, Account Manager gets a fixed-size slice of the client roster. This is stable and non-empty per user across reloads, but **it is not a real permissions filter** — two different real Team Leads could get the same team's data if their user_ids hash the same, and it has no connection to the real `manager_id`/`teamlead_id`/`category_id` hierarchy on the actual `User` model. Swap this for a real query once the ticketing backend has an aggregation endpoint to scope by (same "not real yet" caveat `SuperAdminDashboard`'s own top-of-file comment already carries for the mock dataset in general).

## Permission overrides and permission requests

Two related phases added real, per-user authorization on top of what had always been a purely role-scoped, frontend-only permission system (see "Permission requests" and "Per-user permission overrides" above for the full architectural detail — this is a short retrospective, not a duplicate). First, per-user permission overrides: a new `user_permission_overrides` table, a `PermissionResolverService` merging role defaults with active overrides everywhere effective permissions are computed, a JWT `permissions` claim propagating that merged set to `ticketing-service` (see the root CLAUDE.md), and the first two real backend authorization checks and audit-log writers this service has ever had. Second, permission requests: a self-service request/review workflow addressed to a role rather than a person, deliberately built to delegate the actual grant to the exact same `PermissionOverrideService.grant()` rather than duplicating it, reachable from every role's sidebar via one addition to the existing shared `NAV_ITEMS_BY_ROLE` config rather than a new ticket-workspace-embedded page. `ticketing-service` grew an analogous but independent `ticket:edit_ticket` permission and per-ticket Edit Access request/approve/reject workflow in parallel — the two "request/review" features share a shape but no code, by design (see that repo's CLAUDE.md).

Verified live end-to-end against real dev-database data for every scenario (grant/revoke, request/approve/reject, redundant-grant rejection, Account-Manager-own-reports scoping, Super Admin/Site Lead unconditional authority, JWT claim refresh) rather than just type-checked, with all test-created state cleaned up afterward.
